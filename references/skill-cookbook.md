**遇到问题时查阅：**
铁律一/二 → 第四节（评估体系）→ 第十节（战术手册，按场景索引）
输出质量变差 → 铁律二 + 附录 C（上下文预算）
模型升级后 Skill 失效 → 附录 B（模型版本契约）
评估体系想深入了解 → [附录 D：行业评估实践](附录D-行业评估实践.md)
实操战术详细步骤 → 附录 E（实操战术手册）

**进阶时阅读：**
第七节（分解策略）→ 附录 A（Prompt 工程规范）→ 附录 B/C

**30 分钟上手：完成你的第一个 Skill**

```
1. 发给 Claude Code：
   "请初始化一个新的 Skill 项目，创建 spec/、evals/、src/ 三个目录，
    并初始化版本控制，保存为 v0-baseline。"

2. 准备 2-3 个真实样本（替换敏感信息），
   放入 evals/golden_dataset/（case_01_input.md + case_01_expected.md）。

3. 用最简单的方式写 spec/AGENT_RULES.md：
   只填 3 样东西：我的 Skill 做什么、给它什么输入、期望什么输出。

4. "请基于 spec/AGENT_RULES.md 生成 evals/test_eval.py，
    只需要 3 个最基本的测试用例（正式标准 5 个，上手阶段 3 个即可）。"

   💡 此时运行测试应该全是 FAIL（红灯）——因为还没写代码。
   这恰恰证明测试是有效的。如果没写代码就全绿，说明测试有问题。

5. "请基于 spec/AGENT_RULES.md 在 src/ 下编写 Skill 代码，
    所有修改在隔离分支上进行。"

6. "请运行 evals/test_eval.py，连续运行 3 次确认结果一致。"

7. 3 次都全绿？→ "测试全绿。请保存当前版本并标记为 v1-first-skill。"

8. 没全绿？→ "请修复第一个 FAIL，修完再跑测试。"
```

到这里你已经完成了一个最小可用的 Skill。

## 定位

```
SDD（规格驱动开发）    先定义什么叫成功、什么不许动        ← 地基
EDD（评估驱动开发）    用测试数据验证，而非靠感觉          ← 主体
Agentic Workflow       多角色分工，降低单点复杂度（进阶）  ← 上层建筑
```

**SDD 是 EDD 的前提。** 没有规格定义，测试数据就没有筛选标准；没有测试数据，规格就是一纸空文。三层递进，不跳层。

## 一、三条认知铁律

在学习任何具体方法之前，先内化这三条。它们决定了你面对问题时的第一反应是对还是错。

### 铁律一：Skill 是程序，不是文档

发现 Bug 时，第一反应**绝对不是**在 Prompt 里加一条"请注意XXX"。

**正确反应：** 写一个能稳定复现这个 Bug 的测试用例，加入 evals/test_eval.py。Prompt 里的规则，上下文一长就被 AI 遗忘；写进测试脚本的断言，永远不会失效。

> **没有执行证明的规则，等于不存在。** Prompt 里声明的"红线"或"禁令"，必须落地为 Code-graded eval 的断言或 LLM-judged eval 的打分指标，否则它只是一厢情愿。

### 铁律二：规则模块化，按需加载

输出质量下降时，本能反应是加规则。但往往问题出在**旧规则互相打架**——一条说"要简洁"，另一条说"要详细"，AI 两边都想遵守，结果两边都做不好。

**先做减法或拆分，不做加法：**
- 输出变差了，先检查现有规则有没有互相矛盾的
- 如果矛盾规则中有一条是冗余的 → 直接删
- 如果两条都有业务价值但互相打架 → 拆分到不同模块，按需加载

**规则多了不一定要砍——要拆：**

```
spec/
├── AGENT_RULES.md        ← 核心规则（始终加载）
├── rules_parse.md        ← 解析规则（仅解析任务时加载）
├── rules_tone.md         ← 语气规则（仅润色任务时加载）
└── rules_edge.md         ← 边缘 case 规则（仅排查时加载）
```

**什么时候启动拆分（出现任一条就拆）：**
- 两条规则在同一个任务中给出矛盾指令
- 一条规则只在特定阶段有用，但被所有阶段加载
- 排查后发现质量下降不是规则缺失，是规则太多导致 AI 无法判断优先级

**不设条数硬上限。** 10 条规则如果清晰无冲突，不需要拆；5 条规则如果有冲突，也该拆。以清晰无冲突为准。

**关键区分：删规则，不删示例。** 规则是"你应该怎么做"的指令，示例是"做对了长什么样"的参考。规则多了会打架，示例多了反而帮 AI 理解你要什么。

**给 Claude Code 的排查指令：**
```
"当前输出质量下降了。
在新增任何规则之前，请先检查 spec/ 下的所有规则文件中
是否有相互矛盾或重叠的规则。
列出你发现的矛盾，建议删除或拆分到独立模块。
注意：不要删除任何Few-shot 示例或示例，只审查指令性规则。
如果存在职责混杂（某些规则只在特定阶段有用），
建议按职责拆分为 rules_[职责].md 模块。"
```

**规则模块动态加载指令：**
```
"当前是 [解析/润色/边缘排查] 阶段，请加载以下规则文件：
1. spec/AGENT_RULES.md（核心规则，始终加载）
2. spec/rules_[当前阶段].md（阶段规则）
不需要加载其他阶段的规则模块。"
```

### 铁律三：你的判断力在业务层，不在代码层

输入一个东西，出来的对不对，你最清楚。代码写得优不优雅，不是你的事。

你定义成功标准（业务语言）→ AI 生成代码 → 评估工具验证 → 你做最终取舍。你永远不需要看懂代码。

## 二、项目文件结构

```
项目根目录/
├── spec/                     ← 契约层：定义"什么叫成功"
│   ├── AGENT_RULES.md        ← 成功标准、绝对红线、容忍清单、版本控制规范
│   ├── eval_rubric.md        ← LLM-judged 评分量表 + Few-shot 示例
│   └── api_schema.json       ← 输出结构的机器契约（结构化输出时推荐）
│
├── evals/                    ← 评估层：验证"是否真的成功"
│   ├── test_eval.py          ← Code-graded eval（PASS/FAIL/WARN，不过不能发布）
│   ├── llm_judge_eval.py     ← LLM-judged eval 自动化裁判（进阶，见第四节）
│   ├── diff_tool.py          ← Diff tool（开发辅助，版本间差异比对）
│   └── golden_dataset/       ← Golden Dataset（输入输出 Pair）
│       ├── case_01_input.md
│       └── case_01_expected.md
│
├── src/                      ← 执行层：AI 生成的业务代码
│   └── (skill 的实际代码)
│
└── .git/                     ← 版本控制
```

**关于 src/ 目录：** `src/` 是业务代码目录的泛称。对于 LLM Skill（如 text-smith），业务代码可能是 `modules/ + references/ + SKILL.md`，不一定要叫 `src/`。开发时用 `spec/ evals/` 管规格和评测，发布时 skill 目录结构由 skill 本身决定。三段式的核心是**逻辑分离**（契约、评测、代码），不是**目录命名**。

**三段式命名逻辑：**
- **`/spec`** — 不是说明书，是 AI 必须执行的契约
- **`/evals`** — 统管 Code-graded eval（Python 断言）和 LLM-judged eval（LLM 打分）
- **`/src`** — 代码和契约物理隔离，防止 AI 修代码时顺手改了评估脚本

**关于 api_schema.json：**
你的 Skill 输出能用一个固定的数据结构描述吗？能 → 用 api_schema.json，test_eval.py 直接校验输出是否符合 Schema，比手写断言更精确；不能（自由文本）→ 不用。

**最小示例：**
```json
{
  "type": "object",
  "required": ["title", "segments"],
  "properties": {
    "title": {"type": "string"},
    "segments": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["speaker", "text"],
        "properties": {
          "speaker": {"type": "string"},
          "text": {"type": "string"}
        }
      }
    }
  }
}
```
test_eval.py 可用 `jsonschema` 库校验输出是否符合此 Schema，比手写断言更精确且自动同步。

**给 Claude Code 的初始化指令：**
```
"请按以下结构创建项目目录：
- spec/ 放 AGENT_RULES.md 和 eval_rubric.md
- evals/ 放 test_eval.py 和 golden_dataset/
- src/ 放所有业务代码

同一个版本记录中严禁同时包含 src/ 和 spec/ 或 evals/ 的修改。
结构性变更时，先保存 evals/ 的修改（确认红灯），
再保存 src/ 的修改（确认绿灯），分两次保存。"
```

## 三、spec/AGENT_RULES.md — 规格文件

### 1. 成功基准线（Acceptance Criteria — 怎样算"做对了"）

用业务语言写，只关注"给它什么、出来什么"，不管代码内部怎么实现：

```
【Skill 名称】：_______________
【核心功能】：_______________（一句话）
【输入】：_______________（举一个具体例子）
【期望输出】：_______________（格式 + 内容）
【判定标准】：_______________（怎样算"对了"）
```

### 2. 绝对红线（Hard Constraints）

一旦触发，立即回滚，不继续修补：

- **业务中断** — 崩溃、死循环、跳过核心步骤
- **数据安全** — 覆盖/删除原始数据、数据截断或丢失
- **过度工程** — 未经允许引入新依赖、重构不相关的模块、"顺手"改了别的东西

### 3. 容忍清单（Ignored Warnings — 明确告诉 AI"这些问题不许你修"）

以下问题，AI 修了反而可能引入新 bug，一律不许修：

- □ 代码里有黄色警告线（不影响运行，只是"不够优雅"）
- □ AI 建议"可以用更现代的写法"（现在能跑就行）
- □ 极少出现的奇怪输入导致的小问题（不影响正常使用）
- □ AI 扫描出的"建议性"优化（不修也不影响功能）

看到 AI 建议修这些问题时，回复：*"这个问题在容忍清单里，不需要修。请只关注我要求的修改。"*

### 4. 输出品质要求（Quality Spec）

定义无法用 Python 断言的质量维度（仅当输出含自然语言时需要）：

```
【语气要求】：_______________
【禁止出现】：_______________
【Few-shot 示例】：（贴 1-2 个你认为"对的"输出，最终写入 spec/eval_rubric.md）
```

> **术语说明：** "Few-shot 示例"指单个"做对了长什么样"的参考（推荐 1-2 个，上限 9 个）；"Golden Dataset"指你确认过的完整测试用例库（3-50+ 个）。Few-shot 示例可从 Golden Dataset 的 expected 输出中精选，但二者用途不同——Few-shot 写入 prompt 教 AI 怎么做，Golden Dataset 供测试脚本验证做得对不对。

### 5. 铁规：契约文件必须同步（按变更类型分流）

spec/AGENT_RULES.md、evals/test_eval.py、spec/eval_rubric.md 必须同步（如果项目使用了 api_schema.json，也包含在内）。最大风险不是 bug 修不好，而是**测试本身是错的——全绿但其实没在检测你要的东西**。

但并非所有变更都需要走同一套流程。**按变更性质分流：**

**结构性变更 → 强制 Red-Green 循环：**

增删字段、改步骤、调格式等"骨架"级修改：

```
## 契约文件同步规则 — 结构性变更
- 先修改 evals/test_eval.py，确认新测试在当前代码上会 FAIL（红灯）
- 然后再修改 src/ 下的代码，让测试变 PASS（绿灯）
- 测试改动和代码改动分两次保存版本
- 顺序永远是：改测试 → 确认红灯 → 改代码 → 确认绿灯
```

**语义性变更 → LLM-judged eval 直裁：**

调语气、改人味、调排版等"血肉"级修改，强写 Python 断言是效率浪费：

```
## 契约文件同步规则 — 语义性变更
- 不需要先造红灯
- 修改后直接运行 Skill，将输出交给LLM-judged eval 打分
- LLM-judged eval 所有维度均 ≥ 3 分方可合并
- 仍需在隔离分支上进行
- 调整后必须先重跑 Code-graded eval 确认全绿，再进入 LLM-judged eval
```

**如何判断属于哪类？**

| 你要改的是…… | 分类 |
|-------------|------|
| 增加/删除一个输出字段 | 结构性 |
| 修改处理步骤的顺序 | 结构性 |
| 换输出格式（如 JSON→表格） | 结构性 |
| 改语气（如正式→亲切） | 语义性 |
| 改措辞（如"禁止"→"请勿"） | 语义性 |
| 拿不准？ | 先按语义性做，更简单 |

**给 Claude Code 的指令：**
```
"我要修改 [某功能]。先判断这是结构性变更还是语义性变更：

如果是结构性变更（字段、步骤、格式）：
  先修改 evals/test_eval.py 使当前代码 FAIL，
  确认 FAIL 后再修改主代码让它 PASS。分两次保存。

如果是语义性变更（语气、人味、排版）：
  直接修改主代码，修完后先运行 evals/test_eval.py 确认全绿，
  再将输出交给 LLM-judged eval 打分。"
```

> **Red-Green 循环（先红后绿）适用于结构性变更：** 先让测试变红（FAIL）——证明测试确实在检测这个功能；再改代码让测试变绿（PASS）——证明功能确实修好了。**红灯是测试的测试**——跳过红灯，绿灯可能是假的。语义性变更由 LLM-judged eval 做最终裁决。

## 四、评估体系

```
Grader 类型（回答"对不对/好不好"）
├── Code-graded eval（确定性评估）    确定性检查，PASS/FAIL/WARN
├── LLM-judged eval（语义评估）      语义评分，1-5 分
└── Human-judged eval（人工评估）    人工抽检，最终裁判

Performance metrics（性能指标）       回答"快不快/贵不贵"
├── 延迟（首 token 时间、总耗时）
├── Token 消耗（输入/输出 token 数）
└── 成本（每次调用费用）

开发辅助工具（不是评估，是调试用的）
└── Diff tool（差异比对工具）        版本间差异比对 + 智能分类

封版条件：Code-graded 稳定全绿 + Performance metrics 通过 + LLM-judged 所有维度 ≥ 3 分 + 人工审查
```

**为什么不编号？** 编号暗示串行和层级——先跑 L1 再跑 L2 再跑 L3。但 Code-graded 和 LLM-judged 是并行维度，不是递进关系。一个 task 可以同时挂多个 grader。编号也暗示"上层比下层高级"——但确定性检查和语义评分谁也不比谁高级，它们检查不同维度。

> 行业术语对照：Code-graded eval = Anthropic/OpenAI 说的 "code-based grader"；LLM-judged eval = 行业说的 "LLM-as-judge"；Human-judged eval = "human grader"。详见[附录 D：行业评估实践](附录D-行业评估实践.md)。

### Code-graded eval（确定性评估）— 不过就不能发布

Code-graded eval = 用 Python 脚本自动检查，结果为 PASS / FAIL / WARN。

**做什么：** 只检查"骨架"——格式对不对、字段全不全、步骤有没有跳、有没有崩溃。
**不做什么：** 不检查"血肉"——文字好不好、语气对不对、有没有人味。

> ⚠️ 一旦把"不能出现某个词"之类的语言规则写进 test_eval.py，就会越维护越崩——同一个意思有一百种说法，用代码去卡语言问题是死路。语言问题交给 LLM-judged eval。

**三种结果：**
- **PASS**：通过，无问题
- **FAIL**：不通过，阻塞发布
- **WARN**：黄灯，不阻塞发布但需人工确认。用于带阈值的检查（如保留率 60%-110%，11% 改写幅度是 WARN 而不是 FAIL）

```
python evals/test_eval.py
→ Case 1: PASS
→ Case 2: FAIL (原因：缺少字段X)
→ Case 3: WARN (原因：改写幅度 11.3%，阈值 10%)
→ Case 4: FLAKY (3次中2次PASS，需排查)
```

**封版标准：稳定全绿，而非一次全绿。** AI 输出是概率性的——同样的输入跑两次，输出可能不同。封版前对 test_eval.py **连续运行 3 次**：
- 3 次结果完全一致 → 可以封版
- 出现不一致 → 标记为 FLAKY 用例，进入排查
- 不允许用"多跑几次总有一次过"来封版

**非确定性度量：pass@k / pass^k**（pass^k 由 Anthropic 提出，行业主流仍用 pass@1 作为标准指标）

AI 输出是概率性的，同一个用例跑多次结果可能不同。用两个指标量化：

- **pass@k**：k 次尝试中至少成功 1 次的概率。工具类 skill 用这个。
- **pass^k**：k 次全部成功的概率。面向用户的 skill 用这个。

示例：一个用例单次成功概率 75%，跑 3 次：
- pass@3 ≈ 98.4%（至少成功 1 次）
- pass^3 ≈ 42.2%（3 次全成功）

封版标准：pass^3 = 100%（连续 3 次全成功），而非"跑 3 次至少过 1 次"。

> **为什么是 3 次？** 3 次是检测非确定性的最小次数（2 次无法区分 FLAKY 和偶然失败），5 次提供更高置信度但时间成本翻倍。对大多数 Skill 项目，3 次一致是实用门槛；对高可靠性场景（如金融、医疗），建议扩展到 5 次。

**评结果，不评路径**

Agent 可能用你没预期的方式完成任务。不要因为路径不同就判 FAIL。

示例：发言人标签统一为 `[姓名]` 是对的，统一为 `姓名：` 也是对的，只要全文一致即可。不要硬编码"必须用 `[姓名]`"格式。

如果发现 agent 用了预期外但合理的路径，应该：
1. 更新 grader 以接受这种路径
2. 将新路径作为 Few-shot 示例加入
而不是惩罚 agent 的创造性解法。

**Capability eval vs Regression eval**

| 类型 | 问什么 | 初始 pass rate | 用例选择 | pass rate 变化含义 |
|------|--------|---------------|---------|-------------------|
| Capability eval | "能不能做" | 低（20-50%） | 当前做不好的场景 | 上升 = 进步 |
| Regression eval | "还在不在" | 高（接近 100%） | 当前能做好的场景 | 下降 = 退步 |

关系：Capability eval pass rate 高了可以"毕业"为 Regression eval。Golden Dataset 中的用例应标注属于哪种类型。

**测试可信度自检（首次测试后必做）：**
```
发给 Claude Code：
"请检查 evals/test_eval.py 中每个测试用例：
1. 每个断言是否真的在验证 AGENT_RULES.md 中的要求？
2. 如果把输出改成完全错误的值，这个断言会 FAIL 吗？
   （如果不会，说明断言无效）
3. 有没有测试用例只有注释没有断言？"
```

**FLAKY 用例处理：** 时红时绿的用例不参与 PASS/FAIL 判定。尝试加强 Prompt 约束；仍然 FLAKY → 改由 LLM-judged eval 覆盖。

**FLAKY 的判定标准：** 同一用例连续运行 3 次，结果不完全一致（即非 3/3 PASS 也非 3/3 FAIL），即标记为 FLAKY。例如 3 次中 2 次 PASS、1 次 FAIL，该用例为 FLAKY。FLAKY 用例不参与封版判定，需单独排查原因（常见原因：Prompt 约束不足、模型输出随机性过高、输入边界条件模糊）。

**评估饱和**

100% pass rate 的 eval 只能测回归，不能测进步。当所有 Regression eval 用例都 PASS 时：
1. 不要满足——说明 eval 太简单了
2. 加更难的用例（从生产日志、用户反馈中挖掘）
3. Capability eval "毕业"为 Regression eval，新建更难的 Capability eval

定期检查 eval 是否还有区分度。如果所有版本都 PASS，eval 就没有价值了。

### LLM-judged eval（语义评估）— 打分参考，人做最终判断

LLM-judged eval = 让另一个 AI 对照评分表打分（1-5 分），帮你判断代码判不了的东西。**分数是参考，不是命令——最终你说了算。**

以下是**常见维度示例**，每个项目选取 2-4 个最相关的，不要照搬：

| 维度（示例） | 5分 | 3分（及格线） | 1分 |
|---|---|---|---|
| 人味 | 完全像人写的 | 能用，偶有AI痕迹 | 纯AI输出 |
| 语气 | 完全符合要求 | 基本对路 | 明显不对 |
| 连贯性 | 有清晰推理链 | 段落间有逻辑 | 跳跃、断裂 |
| 冗余度 | 精练，无废话 | 略有冗余 | 大量套话 |

**选择原则：** 说不出为什么需要某个维度，就不要加。数据同步类可能完全不需要 LLM-judged eval，内容生成类可能需要 3-4 个维度。

**LLM-judged eval 的三种执行方式（推荐度递减）：**

**方式一：自动化裁判（推荐，日常迭代用）**

让 Claude Code 生成一个评估脚本 evals/llm_judge_eval.py，自动调用轻量级模型打分：
```
"请编写 evals/llm_judge_eval.py，挂载在 test_eval.py 之后运行。要求：
1. 读取 golden_dataset/ 中的输入，用当前 Skill 生成输出
2. 将输出 + eval_rubric.md + Few-shot 示例发送给轻量级模型 API
3. 按 eval_rubric.md 的维度打 1-5 分
4. 所有维度均 ≥ 3 分则整体 PASS
5. API key 从环境变量读取，不要硬编码"
```

如果设置 API key 遇到困难，**先用方式二手动评估完全没问题**。

**方式二：轻量快捷版（快速检查用）**

即使是轻量快捷版，也应引用项目的实际评分维度，而非凭感觉打分。

在另一个 Claude 对话中：
```
"请按以下维度评价输出的质量（每维度 1-5 分）：
[从 spec/eval_rubric.md 摘取 2-3 个最相关维度]
只给分数和一句话理由。

[贴输出]"
```

**方式三：完整手动评估（封版前用）**

在另一个 Claude 对话中，贴实际输出 + eval_rubric.md + Few-shot 示例，让它打分。所有维度均 ≥ 3 分可封版。

**关键：不要用写代码的同一个 Claude Code 会话做 LLM-judged eval。运动员不能当裁判。**

**防废弃策略：** 日常迭代用方式一（零手动成本）。没搭自动化时用方式二过渡。方式三只在封版前做一次。

**LLM-judged 校准**

LLM-judged 的分数不是绝对真理，需要定期与人工校准：

1. 每 20-50 次自动评估，抽 5 个用例做人工评估
2. 对比人工分数和 LLM 分数，计算一致率
3. 一致率 < 70% → 更新 rubric 或换 judge prompt
4. 一致率 ≥ 70% → 可以信任自动评估

校准数据保存到 evals/reports/calibration/ 目录。

**校准深度指南：**

| 维度 | 建议 | 来源 |
|------|------|------|
| 一致率定义 | 维度级二值一致（同一维度是否都 PASS/FAIL），而非分数完全匹配（3 vs 4 不算不一致） | Eugene Yan |
| Judge 模型选择 | 用与 Skill 不同的模型做 judge；用最强可用模型；给 judge 加 chain-of-thought（先分析再打分） | OpenAI, Anthropic |
| 位置偏差 | 成对比较时交换顺序跑两次，结果不一致则标 tie | Eugene Yan |
| 二值 vs 刻度 | 日常迭代用 pass/fail（标注一致性更高）；封版前用 1-5 分（获取细粒度反馈）。刻度评分的 3 vs 4 分歧无法解决时，降级为二值 | Eugene Yan |
| 退出选项 | 给 LLM judge 一个"无法判断"选项，避免幻觉评分 | Anthropic |
| 领域差异 | 通用领域一致率约 85%，专业领域降至 60-68%。专业领域需更频繁校准 | DataVLab 2026 |

**校准记录格式示例：**
```
evals/reports/calibration/2026-05-07.md
- 日期：2026-05-07
- Judge 模型：claude-sonnet-4-6
- 抽样：5 个用例（case_03, case_07, case_12, case_18, case_25）
- 一致率：4/5 = 80%（维度级二值一致）
- 不一致用例：case_12（LLM 评 4 分，人工评 2 分，原因：judge 未识别出术语错误）
- 行动：在 rubric 中增加术语准确性维度的评分标准
```

### Human-judged eval（人工评估）— 最终裁判

人工评估是金标准，但贵且慢。主要用于：
- **校准 LLM-judged eval**：定期对比，确保自动打分可靠
- **封版前终审**：所有自动化检查通过后，人工抽检关键用例
- **争议裁决**：Code-graded 和 LLM-judged 结果矛盾时，人工判断

**人工评估的频率：** 不需要每次都做。日常迭代靠 Code-graded + LLM-judged，封版前做一次人工抽检，每 20-50 次自动评估做一次校准。

### Performance metrics（性能指标）— 跑得快不快

Code-graded eval 只管"对不对"，不管"快不快"。10 行数据秒过、10 万行跑 10 分钟的情况，Code-graded 完全感知不到。

**性能指标三类，不需要每次全做：**

**① 静态预判（绿线阶段就开始做）：**

```
发给 Claude Code：
"请分析当前 Skill 的性能预判指标：
1. 统计当前加载的所有规则文件的总 token 数（估算）
2. 列出流程中是否有重复调用的节点
3. 检查是否有中间结果被重复生成
4. 当前阶段是否加载了不需要的规则模块
5. Few-shot 示例数量是否超过必要限度
如果任何一项有问题，给出具体的精简建议。"
```

**② 运行时断言（黄线阶段加入）：**

在 test_eval.py 中增加性能基线——首次封版时记录实际耗时，后续封版检查是否退化：

```
发给 Claude Code：
"请在 evals/test_eval.py 中增加一个性能测试用例：
记录当前 Skill 处理标准测试输入的耗时作为基线，
后续运行时如果耗时超过基线的 1.5 倍，标记为 FAIL。"
```

**③ 规模压测（蓝线封版前做）：**

```
发给 Claude Code：
"请用以下三种数据规模分别运行 Skill，记录每次的耗时：
1. 10 条数据（基线）
2. 100 条数据
3. 1000 条数据
判断耗时增长是线性的还是超线性的。
如果是超线性增长，分析瓶颈在哪里。"
```

**什么时候做：** 绿线只做①；黄线加入②；蓝线封版前做③。

**④ 成本治理（蓝线阶段开始追踪）：**

延迟和 token 消耗是性能的"快不快"维度，成本是"贵不贵"维度。两者相关但不等价——模型路由可以降成本而不降延迟。

```
绿线阶段：
在 spec/AGENT_RULES.md 顶部声明单次调用成本基线：
"单次调用成本基线：¥___ / $___（含输入 + 输出 token）"

黄线阶段：
在 evals/test_eval.py 中增加成本预警——
单次调用成本超过基线的 1.5 倍，标记为 WARN。

蓝线阶段：
评估以下成本优化策略：
1. 模型路由：60-70% 的任务可用小模型完成，按任务难度路由
2. 语义缓存：相似输入命中缓存，命中率通常 30-40%
3. Batch API：非实时任务使用批处理端点，成本降低约 50%
4. Prompt 精简：移除冗余规则和示例，每节省 1K token ≈ 节省 3-5% 成本
```

**成本预算上限：** 单任务最大花费 = 3 × 中位数成本。超出时停止执行并报警，防止重试循环消耗无上限预算。

**给 Claude Code 的指令：**
```
"请分析当前 Skill 的成本情况：
1. 统计最近 10 次调用的平均 token 消耗（输入/输出分别统计）
2. 计算单次调用成本（按当前模型定价）
3. 是否存在可缓存重复计算的节点
4. 是否有规则模块在不需要时仍被加载（浪费输入 token）
5. 给出成本优化建议，按节省幅度排序。"
```

### Diff tool（差异比对工具）— 开发辅助，不是评估

Diff tool = 对比两个版本的输出差异，自动分类已知差异类型，标注需要人工确认的变更。

**它不是 grader**——它不判 PASS/FAIL，不判 1-5 分。它是开发调试时的"眼睛"：改完 skill 后，第一时间看"到底改了哪些地方"。

**什么时候用：**
- 修改 skill 代码后，对比改前改后的输出差异
- 新增 Few-shot 示例后，检查影响范围
- 升级模型后，快速定位行为变化

**自动分类示例：**
```
python evals/diff_tool.py --case all
→ 填充词删除: 12 处 (自动确认)
→ ASR 纠错: 8 处 (自动确认)
→ 发言人校准: 5 处 (自动确认)
→ 需人工确认: 19 处
```

**和 grader 的关系：** Code-graded eval 回答"合不合格"，LLM-judged eval 回答"好不好"，Diff tool 回答"改了什么"。三个工具互补，不互相替代。

## 五、版本控制安全网（金丝雀发布）

> 金丝雀发布（Canary Release）：矿工下井前放一只金丝雀试探毒气——AI 每次改代码也一样，先在隔离的安全分支上试，确认没问题再合并到正式版本。
> **绝对禁止在正式版本（main）上直接修 bug。**

> 每个操作使用**自然语言指令**（发给 Claude Code）。底层 Git 命令见附录 E 的 Git 命令速查表。

**项目初始化（只做一次）：**
```
发给 Claude Code：
"请初始化项目的版本控制，把当前所有文件保存为初始版本，
标记为 v0-baseline。"
```

**每次修 bug 前：**
```
"请创建一个隔离的工作分支，命名为 fix/[问题简述]。
在这个分支上修复，修完后运行 evals/test_eval.py。"
```

**修完后的三种情况：**

```
情况 A：test_eval.py 全 PASS（连续 3 次一致）
→ "测试全部通过。请将这个分支合并到主线版本，
   打上版本标签 v[N]-[描述]，然后删除工作分支。"

情况 B：有 FAIL，但没引入新错误
→ "继续在当前分支修复第一个 FAIL，不要动其他东西。"
  ⚠️ 同一个 FAIL 连续修复 3 次仍未 PASS → 按情况 C 处理

⚠️ **当情况 B 和情况 C 同时满足时，情况 C 优先**——先止损（放弃分支），再判断是否换路径重新解决原问题。引入新 FAIL 的破坏性大于旧 FAIL 未修好。

情况 C：修复导致 2 个以上新的 FAIL
→ 先排除非确定性（对失败用例单独跑 3 次，3 次都 FAIL 才确认）。
  确认后立即止损：
  "这次修复引入了新问题，放弃这个分支。
   回退到主线版本。
   创建一个新分支，用完全不同的技术路径重新解决原问题。"

  回退后，先让 AI 说思路再动手：
  "回退到主线后，请分析上一次修复失败的原因，
   告诉我你打算用什么不同方法解决。不要开始写代码。"
```

### 为什么情况 C 要强制换路径？

AI 在烂摊子上继续修补，只会越补越烂——它会一直沿着错误的方向打补丁。强制回到干净版本、换一种完全不同的方法重做，反而比死磕更快。

### 给 Claude Code 的全局指令（写入 spec/AGENT_RULES.md）

```

- 所有修改必须在隔离分支上进行，严禁直接修改主线版本
- 每次保存版本前必须运行 evals/test_eval.py
- 如果修复导致超过 2 个新的 FAIL，立即停止并报告
- 严禁在失败的分支上继续叠加修复
- 同一个 FAIL 连续修复 3 次仍未 PASS，放弃当前分支，换路径
```

## 六、Golden Dataset（黄金数据集）— 你确认过的"标准答案库"

### 什么是 Golden Dataset

你亲手确认过的"输入→理想输出"配对，代表真实业务场景。不是 AI 生成的，是你自己看过、认可的。

```
evals/golden_dataset/
├── case_01_input.md       ← 真实业务输入
├── case_01_expected.md    ← 你确认过的理想输出
└── ...
```

### Few-shot 示例 vs Golden Dataset

这是两个不同的东西，不要混淆：

| | Few-shot 示例 | Golden Dataset |
|---|---|---|
| 给谁看 | 给 AI（写入 prompt） | 给测试脚本（验证输出） |
| 数量 | 推荐 1-2 个，上限 9 个（受 token 预算限制） | 3-50+ 个，越多越好 |
| 作用 | 教学——"这样做是对的" | 考试——"这样做是否通过" |
| 位置 | `spec/eval_rubric.md`（评分量表内） | `evals/golden_dataset/` |
| 类比 | 教科书例题 | 期末考试卷 |

v5 的"金标准样本"一词有歧义（既指 Few-shot 示例又指 Golden Dataset 参照）。v5.5 统一如下：
- **Few-shot 示例** = 给 AI 看的教学样本（写入 prompt）
- **Golden Dataset** = 完整的测试用例库（给测试脚本用）
- **Golden Dataset 中的 expected 输出** = 验证参照（不另造词）

### 规模建议

| 阶段 | 数量 | 用途 | 毕业条件 |
|---|---|---|---|
| 绿线 | 3-5 个 | 验证核心功能 | Code-graded eval 稳定全绿 |
| 黄线 | 10-20 个 | 覆盖主要变体 | LLM-judged eval 所有维度 ≥ 3 分 |
| 蓝线 | 20+ 个 | 回归测试 + 对抗样本 | 包含 ≥ 5 个对抗样本，全 PASS |

### 反向用例（不该触发的场景）

只测"应该触发"不测"不应该触发"，会造出一个过度触发的 agent。

示例：
- 测试"应该删除填充词"的同时，也要测"不应该删除语义连接词"
- 测试"应该纠错 ASR"的同时，也要测"不应该修改正确的文本"
- 测试"应该统一发言人格式"的同时，也要测"不应该改变发言内容"

Golden Dataset 中正向用例和反向用例比例建议 3:1 到 5:1。

### 如何积累

**绿线阶段：** 自然生长。跑通一次，好的输出存下来。

```
"请把刚才的输入保存为 evals/golden_dataset/case_0X_input.md，
 输出保存为 evals/golden_dataset/case_0X_expected.md。"
```

**黄线阶段：** 不能只靠自然积累，必须主动设计对抗样本：

```
"请基于 AGENT_RULES.md 的成功基准线，
为 evals/ 设计 5 个对抗性测试用例：
1 个空输入、1 个超长输入、1 个格式错误、
1 个含特殊字符、1 个语言混杂。
每个用例只要求不崩溃、不丢数据，不要求正确输出。"
```

**从生产日志挖掘：** 用户反馈的 bug、线上监控到的异常，都是天然 eval 用例。每次遇到问题，第一反应是写一个 eval 用例，而不是只改代码。

### 与 test_eval.py 的关系

test_eval.py 从 Golden Dataset 中抽取用例。Dataset 扩大了，test_eval.py 跟着扩展。Dataset 是源头，test_eval.py 是它的可执行形式。

## 七、复杂 Skill 的分解策略

### 判断标准（出现任一条就拆）

- test_eval.py 需要验证三种以上完全不同的能力
- 修一个功能经常破坏另一个功能
- AGENT_RULES.md 的成功基准线超过三条

### 拆分方式

不需要搭多智能体架构。**把一个大 Skill 拆成几个小 Skill，串行调用：**

```
之前：skill_monolith/          一个 skill 做所有事

之后：
skill_parse/      ← 只负责解析输入
skill_transform/  ← 只负责数据转换
skill_render/     ← 只负责生成输出（"人味"在这里）
```

每个小 Skill 有自己的 spec/AGENT_RULES.md 和 evals/test_eval.py。

### 拆分后的管线集成测试

每个子 Skill 单独绿 ≠ 管线全绿。拆分后必须增加端到端验证：

```
"请用第一步的真实输出喂给第二步，验证端到端管线不中断。"
```

### 进阶：多角色协作（Agentic Workflow）

当拆分后的小 Skill 都调稳了，再考虑用 Agent 编排：

```
Router Agent   → 判断意图，分发任务
Worker Agent   → 执行硬核逻辑
Judge Agent    → 对 Worker 输出打分，不过就自动重试（最多 3 次）
Polisher Agent → 注入"人味"，润色最终输出
```

**跃迁信号：** 子 Skill 之间的衔接逻辑复杂到需要手动编排调用顺序时，引入 Router Agent。

**成本警示：** 多 Agent 架构的成本远高于单 Skill。行业数据显示，3 个 agent 的串行管线消耗约 29K tokens，等效单 agent 仅需 10K tokens（约 3x）。Judge Agent 的重试机制会进一步放大成本（最多 3 次重试 = 最多 4 倍 Judge 调用）。仅当单 Skill 的维护成本（频繁修 bug、规则冲突、质量不稳定）显著超过多 Agent 的运行成本时，才考虑拆分。

**这是毕业阶段的架构，不是入门阶段的要求。**

## 八、分层推进策略

```
⚫ 安全层    版本控制初始化、测试环境就绪、绝对红线生效
  ↓
🟢 绿线     Code-graded eval 稳定全绿（连续 3 次一致）
  ↓
🟡 黄线     LLM-judged eval 所有维度均 ≥ 3 分
  ↓
🔵 蓝线     Golden Dataset ≥ 20 且包含 ≥ 5 个对抗样本，全 PASS
  ↓
🟣 紫线     复杂 Skill 拆分完成，子 Skill 各自达到黄线
  ↓
🔴 极线     平台限制 workaround、极端边缘 case
```

**每一层稳了才进下一层。**

### 各层级毕业条件

| 层级 | 毕业条件 |
|------|---------|
| ⚫→🟢 | Code-graded eval 稳定全绿（连续 3 次一致） |
| 🟢→🟡 | LLM-judged eval 所有维度均 ≥ 3 分 |
| 🟡→🔵 | Golden Dataset ≥ 20，包含 ≥ 5 个对抗样本，全 PASS |
| 🔵→🟣 | 触发拆分条件时拆分完成，子 Skill 各自达到黄线 |
| 🟣→🔴 | 主流程 + 子 Skill 连续 5 次修改无新 FAIL |

> **极线 ≠ 绝对红线。** 极线是最高进阶阶段（处理平台限制和极端边缘 case），绝对红线是不可触碰的禁令（数据丢失、业务中断等）。两者完全不同。

### 成熟度梯度：控制随信心递减

```
初期（不信任期）    → 重度管控：每次只改一个文件，注释骨架，拒绝重构
中期（建立信心期）  → 适度放权：允许 AI 在单个模块内自由实现，保持评估验收
成熟期（信任期）    → 轻量治理：主要靠 Golden Dataset 回归测试，规则精简到核心几条
```

**判断标准：** 连续 5 次修改都没触发新 FAIL → 考虑松开一层管控。管控的目的是建立信任，不是永远锁死。

## 九、工作流速查

### 每次调试

```
1. 创建隔离分支 fix/[问题]
2. 修复第一个 FAIL（只修一个）
3. 运行 evals/test_eval.py（连续 3 次确认稳定）
3.5 有 FAIL → 根据检查项定位问题文件（见下方诊断映射），不要盲目修改
4. 全 PASS → 合并到主线版本，打标签
5. 有 FAIL 但没有新增 → 继续修（同一 FAIL 超 3 次 → 换路径）
6. 新增 ≥ 2 个 FAIL → 排除非确定性后确认，放弃分支，换路径
7. Code-graded eval 稳定全绿 → 做 LLM-judged eval
8. LLM-judged eval 所有维度均 ≥ 3 分 + 人工审查 → 封版
```

### FAIL 诊断映射

运行 evals/test_eval.py 后有 FAIL 时，根据检查项定位问题文件：

| FAIL 检查项 | 可能的问题文件 | 排查方向 |
|-------------|---------------|---------|
| 格式/字段缺失 | spec/AGENT_RULES.md | 成功基准线或输出格式定义不完整 |
| 禁止词出现 | spec/AGENT_RULES.md 或 references/ | 禁令规则遗漏或 Few-shot 示例冲突（注：禁止词检查应走 LLM-judged eval，不应写进 test_eval.py） |
| ASR 残留 | references/ 中的清洗规则 | ASR 纠错规则不覆盖当前错误类型 |
| 填充词残留 | references/ 中的清洗规则 | 填充词列表不完整 |
| 保留率异常 | src/ 中的核心逻辑 | 质量门阈值或删除逻辑过度/不足 |

> 不要盲目修改。先确认是规则问题还是测试用例问题——测试用例本身也可能过时。

### 给 Claude Code 的指令模板

**调试时：**
```
请基于 spec/AGENT_RULES.md 测试当前代码。
1. evals/test_eval.py 全 PASS 且没触发绝对红线
   → 回复"契约达成，可以封版"，严禁提优化建议
2. 只有触发绝对红线时才能报告错误，用最少代码修复，严禁重构
3. 所有修改在隔离分支上进行，只改 src/ 下的文件
4. 如果修复导致超过 2 个新 FAIL，立即停止并报告
5. 同一个 FAIL 连续修复 3 次仍未 PASS，停止并报告
```

**开发新 Skill 时：**
```
请先基于 spec/AGENT_RULES.md 生成：
1. evals/test_eval.py（5 个用例，PASS/FAIL）
2. spec/eval_rubric.md（评分量表 + Few-shot 示例）
两个文件经我确认后，才能在 src/ 下写主干代码。
所有开发在隔离分支上进行。
```

## 关键认知

**三条铁律（最重要）：**
1. **Skill 是程序，不是文档。** 发现 bug 第一反应是写测试用例，不是改 prompt。没有执行证明的规则等于不存在。
2. **规则模块化，按需加载。** 质量下降时先删矛盾规则或拆分到独立模块，不做加法。不设条数硬上限，以清晰无冲突为准。
3. **你的判断力在业务层。** 规格你来定，评估数据你来确认，代码让 AI 写。你永远不需要看懂代码。

**AI-Native 要点：**
4. **AI 输出是概率性的。** 封版标准是"稳定全绿"（连续 3 次一致），不是"一次全绿"。FLAKY 用例不参与判定。
5. **先跑通，再质量，最后效率。** Code-graded eval 管骨架（能不能跑），LLM-judged eval 管血肉（质量好不好），性能评估管速度（跑得快不快）。
6. **性能不是事后优化，是三层递进。** 静态预判 → 运行时断言 → 规模压测。绿线只做预判，黄线加断言，蓝线封版前压测。
7. **Red-Green 循环只用于结构性变更；语义性变更由LLM-judged eval 直裁。** 改字段、改步骤 → 先红后绿；改语气、改人味 → LLM-judged eval 打分。Prompt 的结构性变更（增删指令、改步骤顺序）视为结构性变更走 Red-Green 循环；Prompt 的语义性变更（调语气、改措辞）仍按语义性流程处理；多 Prompt 模块使用时必须做组合测试（见附录 A）。
8. **LLM-judged eval 自动化优先。** 日常用自动化裁判，封版前做完整评估，避免手工搬运导致评估被废弃。

**方法论要点：**
9. **SDD 是地基，EDD 是主体，Agentic Workflow 是上层建筑。** 三者递进，不跳层。
10. **契约文件必须同步。** 结构性变更分两次保存；语义性变更先 Code-graded eval 确认再 LLM-judged eval 打分。
11. **金丝雀发布：AI 永远在隔离分支上工作。** 新增 ≥ 2 个 FAIL 就放弃分支、换路径。同一 FAIL 超 3 次也换。
12. **Golden Dataset 是生长出来的。** 绿线阶段自然积累，黄线阶段主动设计对抗样本。
13. **复杂 Skill 调不稳，先拆分再调试。** 一个 Skill 只做一件事，拆分后做端到端管线测试。
14. **容忍清单防止 AI 擅自优化不该修的东西。** 每个项目都要明确"什么不修"。
15. **成熟度梯度。** 初期重度管控 → 中期适度放权 → 成熟期轻量治理。管控的目的是建立信任，不是永远锁死。

## 附录 A：Prompt 工程规范（进阶）

当项目成熟后，业务规则和 Prompt 指令应分离管理：

```
spec/
├── AGENT_RULES.md        ← 业务规则（人写的，定义成功标准）
├── prompts/              ← Prompt 库（AI 读的，驱动行为）
│   ├── system.md         ← 系统 Prompt
│   ├── rules_parse.md    ← 解析阶段 Prompt
│   └── CHANGELOG.md      ← Prompt 变更日志
```

初期不需要拆分。当触发铁律二的拆分信号时，连同 Prompt 库一起拆。

**Prompt 变更日志（CHANGELOG.md）：** 每次修改 Prompt，记录改了什么、为什么改、评估结果。

**Prompt 变更走 Red-Green 循环：** Prompt 的结构性变更（增删指令、改步骤顺序）走 Red-Green 循环；Prompt 的语义性变更（调语气、改措辞）按语义性变更流程处理。

**Prompt 组合测试（使用多个 Prompt 模块时必做）：**

当 Skill 使用多个 Prompt 模块时，必须测试组合效果——单个模块没问题不代表组合后没问题：

```
"请在以下三种配置下分别运行 Skill 并对比输出：
1. 只加载 rules_parse.md
2. 只加载 rules_tone.md
3. 同时加载 rules_parse.md + rules_tone.md
对比 3 次输出，检查组合后是否出现指令冲突。"
```

## 附录 B：模型版本契约 / Model Pinning（进阶）

模型升级后 Skill 可能全面失效。在 spec/AGENT_RULES.md 中声明（行业也称 "model pinning" 或 "model snapshot"）：

```
## 模型契约
- 目标模型：[模型名称及版本]
- temperature：0（确定性优先）
- 最后一次全量验证日期：[日期]
```

**模型升级时：** 冻结所有代码和 Prompt，只换模型，跑全量 test_eval.py。全绿 → 更新版本号封版。有红 → 最小适配，不重写。

**规则的模型相关性标注：**
```
- "输出 JSON 格式" → [模型无关] 所有主流模型都能遵从
- "不要使用列表格式" → [模型相关] 部分模型会忽略否定指令
- "用同事口吻" → [模型相关] 高度依赖模型对语气的理解
```

模型升级时优先重测模型相关规则。

**模型升级的影子测试：**

金丝雀发布适用于代码变更。模型升级需要更强的安全网——影子测试：

1. 冻结所有代码和 Prompt，只换模型
2. 将生产输入同时发送到新旧模型，比较输出差异
3. 用 Diff tool 自动分类差异，人工确认关键变更
4. 差异在容忍范围内 → 灰度发布（1% → 10% → 100%）
5. 差异超出容忍范围 → 最小适配，不重写

影子测试的容忍范围由 Golden Dataset 的 expected 输出定义——不是"输出完全一致"，而是"核心字段和语义不变"。

## 附录 C：上下文预算规范（进阶）

规则模块化解决了冲突问题，但没有解决容量问题。

**上下文预算分配（在 spec/AGENT_RULES.md 顶部声明）：**

```
## 上下文预算（模型：Claude Sonnet，窗口：200K tokens）
| 分区 | 预算 | 实际 |
|------|------|------|
| System Prompt + 规则 | ≤ 8K | ___ |
| Few-shot 示例 | ≤ 4K | ___ |
| 用户输入（单次最大） | ≤ 100K | ___ |
| 输出预留 | ≥ 20K | ___ |
| 缓冲区 | ≥ 8K | ___ |
```

任何规则新增时，必须在此表中更新实际占用量。若实际 > 预算，必须删减或拆分。

**上下文压缩原则：**
1. 规则用祈使句，不用解释性语句。 ✅ "输出 JSON，不要 Markdown"  ❌ "因为下游系统需要 JSON 格式，所以请……"
2. 示例优于规则。一个 50 token 的示例 > 一条 200 token 的规则
3. 规则模块只在需要时加载，用完即卸载

**上下文健康检查：**
```
"请报告本次运行的上下文使用情况：
- 输入总 token 数（估算）
- 规则模块实际加载了哪些
- 是否有规则被输入数据挤出了有效注意力范围"
```

**上下文质量标准（Context Engineering 5 维度）：**

除了预算管理，上下文还需要满足以下 5 个质量标准：

| 维度 | 含义 | 与铁律二的关系 |
|------|------|---------------|
| Relevance（相关性） | 只加载当前阶段需要的规则模块 | 铁律二"按需加载"的直接体现 |
| Sufficiency（充分性） | Few-shot 示例覆盖主要场景 | 示例多了帮 AI 理解，规则多了打架 |
| Isolation（隔离性） | 不同 Skill/Agent 的上下文不交叉污染 | 拆分 Skill 时尤其重要 |
| Economy（经济性） | 总 token 在预算内 | 附录 C 上下文预算的量化体现 |
| Provenance（来源可溯） | 每条规则可追溯到 spec/ 中的定义 | 防止"幽灵规则"——没人知道为什么加的 |

**给 Claude Code 的指令：**
```
"请检查当前 Skill 的上下文质量：
1. Relevance：是否有规则模块在当前阶段不需要但仍被加载？
2. Sufficiency：Few-shot 示例是否覆盖了主要场景和边缘 case？
3. Isolation：不同 Skill/Agent 的规则是否混在同一个文件中？
4. Economy：总 token 是否在预算内？
5. Provenance：每条规则是否都能追溯到 spec/ 中的定义？"
```
