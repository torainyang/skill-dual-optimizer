# Skill 设计评审框架

这份框架用于帮助 `skill-optimizer` 判断目标 skill 的设计模式是否合理。

它融合了两类方法论：

1. **Anthropic / Claude Code 的实战经验**
   - skill 不只是 Markdown 文件，而是一个文件夹
   - description 是给模型看的触发规则
   - Gotchas 是高信号区域
   - 要善用渐进式披露、脚本、模板、hooks、setup/config、memory、分发等能力

2. **Google Cloud 总结的 5 种 Agent Skill 设计模式**
   - Tool Wrapper
   - Generator
   - Reviewer
   - Inversion
   - Pipeline

---

## 一、先判断目标 skill 属于哪种模式

先不要急着改写。先判断当前 skill **最主要**属于哪一种模式，必要时可标注"主模式 + 次模式"。

### 1. Tool Wrapper

特征：
- 主要价值是给模型提供某个库、产品、系统或领域的按需知识
- 常配合 `references/` 使用
- 更强调"什么时候触发"和"读哪些资料"，而不是复杂流程

适合场景：
- 文档知识库
- SDK / API / CLI 用法说明
- 内部平台知识注入

高频问题：
- description 太弱，导致触发不到
- 知识全堆在 `SKILL.md`，没有拆到 `references/`
- 写了很多常识，没有写真正的 gotchas

### 2. Generator

特征：
- 主要价值是稳定产出结构化结果
- 常配合 `assets/` 模板和 `references/` 风格指南
- 更强调输出一致性

适合场景：
- 文档生成
- 报告生成
- 模板化写作
- 结构化脚手架

高频问题：
- 输出结构不固定
- 模板应该拆到 `assets/` 却还塞在 `SKILL.md`
- 风格规则和流程规则混在一起

### 3. Reviewer

特征：
- 主要价值是评审、诊断、打分、发现问题
- 通常先判断"哪里不对"，再给建议
- 常适合配 checklist / rubric

适合场景：
- skill 审查
- 配置评审
- 代码 review
- 质量检查

高频问题：
- 只给结论，没有评价维度
- 审查标准分散，不成体系
- 容易直接跳到修改，绕过审查本身

### 4. Inversion

特征：
- 目标不是立刻生成，而是先提问、先收集信息
- 适合存在关键输入缺口的任务
- 要求明确门槛：信息不足时禁止直接产出最终结果

适合场景：
- 项目规划
- 需求澄清
- 高定制设计
- 复杂配置方案

高频问题：
- 应该先问却没问
- 缺少"未收集完信息前不得继续"的硬门槛
- 一上来就输出大而全方案

### 5. Pipeline

特征：
- 任务必须按阶段顺序执行
- 常有检查点和阶段门槛
- 强调不能跳步

适合场景：
- 多步骤工作流
- 下载 → 分析 → 生成 → 审核 → 交付
- 高风险操作流程

高频问题：
- 步骤顺序不清楚
- 缺少阶段门槛
- 用户未确认就进入下一步
- 工作流过长却没有拆分参考文件或脚本

## 二、再判断模式是否匹配

识别完模式后，继续判断：

- 当前模式是否适合这个 skill？
- 是否本该是 Generator，却写成了杂糅的大说明书？
- 是否本该是 Inversion，却没有提问门槛？
- 是否本该是 Pipeline，却没有检查点？
- 是否其实是多个模式组合，但主次关系不清？

输出时可以这样表达：

- **主模式：Pipeline；次模式：Generator**
- **问题：当前 skill 实际是 Pipeline，但没有显式阶段门槛，导致执行稳定性不足**

---

## 三、常见误判提醒

- 不要为了套模式而套模式
- 不要一看到 skill 很长，就默认必须拆文件
- 不要把"用户只想微调 description"误做成整 skill 重构
- 不要因为 references 存在，就默认渐进式披露已经做得好
- 不要把"有流程"误认为"有门槛"
- 不要把"有脚本"误认为"设计就合理"
