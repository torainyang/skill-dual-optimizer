## 两维度诊断框架

### D1：上下文与执行可靠性

LLM 能不能稳定执行这个 skill？

**注意力经济学**：LLM 对 system prompt 的注意力分布不均匀——前 ~20% 和后 ~20% 是高注意力区，中间 ~60% 是低注意力区（首因-近因效应）。

- ✓ 关键约束在 SKILL.md 首尾 / ✗ 最重要的规则被埋在中间60%
- ✓ Gotchas 在高注意力位置 / ✗ Gotchas 放在工作流中间

**中间遗忘**：规则遗忘有两个原因——分散和距离。分散指同一规则在多处定义；距离指规则定义点和执行点相隔太远。

- ✓ 规则唯一定义点，执行点一句话引用 / ✗ 同一规则在 3 处完整定义
- ✓ 定义点距执行点 <20 行 / ✗ 定义点在 Step 2，执行点在 Step 5

**信噪比**：运行时指令和参考知识的边界。SKILL.md 应只放运行时指令（LLM 执行时必须知道的内容），方法论和评估标准放 references。

- ✓ SKILL.md 只放运行时指令 / ✗ 评分标准和方法论混在 SKILL.md
- ✓ 按需加载触发条件覆盖所有使用场景 / ✗ 核心流程依赖的文件无法被触发加载

**执行稳定性**：步骤依赖是否显式声明，中间产物格式是否明确，分支是否都有出口。

- ✓ 每步声明输入/产出 / ✗ Step 2 的产出格式未定义，Step 3 不知道消费什么
- ✓ 每个条件分支都有明确出口 / ✗ Quick-Fail 后不知道该去哪

**字数效率**：>5000 字警惕，>10000 字必须精简。注意：LLM 做定性判断（明显超标/明显精简/中间区），精确字数统计用脚本。

- ✓ <5000 字或合理使用 references / ✗ 8000 字全堆在 SKILL.md，未拆分
- ✓ 方法论和分类学放 references / ✗ 模式分类6种完整定义占主文件 10%

**引用文件健康**：文件内无增量重复，与 SKILL.md 不重复（执行点强化和红线重申除外），脚本 vs 参考分明。

- ✓ 每个引用文件有独立价值 / ✗ 引用文件与 SKILL.md 大段重复

### D2：认知与决策质量

Skill 的输出真正有用吗？

**推理衰减**：靠后步骤的可靠性会下降。即使上下文充分，指令累积和模式锁定也会导致衰减。

- ✓ 后 1/3 步骤有新硬约束 / ✗ 后半段只是重复前半段的模式
- 启发式检查：后 1/3 步骤是否有新的判断标准？若无，推理可能衰减

**幻觉与遵从**：幻影指令让 LLM 做不可执行的事（如"默默验证""精确打分""计算 token 数"）；负向引导让坏示例反而增加坏输出概率。

- ✓ 每条指令都可执行 / ✗ "输出前先在心中验证"——LLM 无法"心中验证"
- ✓ 好/坏示例比 ≥ 3:1 / ✗ Gotchas 只有坏示例没有好示例

**LLM vs 脚本边界**：需要语义理解 → LLM；确定性操作 → 脚本。高频误配：字数统计、token 计数、格式校验、文件合并。

- ✓ LLM 做判断，脚本做计算 / ✗ 让 LLM 精确统计字数
- ✓ "判断内容是否冗余" → LLM + 限定判断框架 / ✗ "判断内容是否冗余" → 全推给脚本

**防护完备性**：确认门槛是否硬编码，Gotchas 是否覆盖高频错误。

- ✓ "等待用户确认后进入下一步" / ✗ "建议确认"（软约束）
- ✓ Gotchas 每条有正/负示例 / ✗ Gotchas 裸奔（只有规则没有示例）

**输出可执行性**：用户拿到输出后能否直接行动，还是需要二次加工？

- ✓ 输出格式稳定，字段完整 / ✗ 输出有时是列表有时是段落，用户需要重新整理

## 专家组诊断方法论

### 何时用专家组诊断

- 首次审查一个复杂 skill
- 框架诊断发现 P0 问题但根因不明
- 审查 skill-dual-optimizer 自身（自指问题需要外部视角）
- 用户要求"深度审计"

### 专家角色选择原则

- 覆盖核心矛盾面（安全 vs 体验、短期 vs 长期、结构 vs 内容）
- 每个专家有独立评估维度，不重叠
- 数量取决于复杂度：简单 2 个，典型 3 个，复杂 4-5 个

### 推荐专家组合

| Skill 模式 | 推荐专家组合 |
|-----------|------------|
| Generator | Prompt 架构师 + 认知负荷分析师 |
| Pipeline | 流程完备性审计师 + 认知负荷分析师 |
| Reviewer | Prompt 架构师 + 自指逻辑审计师 |
| Hybrid | Prompt 架构师 + 认知负荷分析师 + 流程完备性审计师 |
| Tool Wrapper | Prompt 架构师 + 认知负荷分析师 |

### 四个专家角色的审计维度

**Prompt 架构师**：注意力分布、幻影指令、负向引导风险（好/坏示例比）、矛盾指令、运行时/元数据分离

**认知负荷分析师**：字数效率（各区块占比）、信息架构（运行时 vs 参考知识边界）、渐进式披露（按需加载逻辑）、中间遗忘（定义点与执行点距离）、重复分析（SKILL.md 与 references 之间）

**流程完备性审计师**：步骤依赖与中间产物格式、边界条件覆盖（多目标/中途变向/文件缺失）、决策树缺口（Quick-Fail 后路径/模式分类穷举性）、Gotchas 缺口、确认门槛遗漏、Verify 完备性

**自指逻辑审计师**（建议由主 agent 执行，不做子 agent 递归）：D1/D2 逐条自指验证、Quick-Fail 自指、references 一致性。注意：自指审计只做一轮，不做递归。

### 执行规则

1. 启动 N 个并行 agent，每个 agent 角色和视角由指定或自动推断
2. 共享输入（目标 skill 全文 + references），独立输出（互不可见）
3. 汇总时按"共识 → 分歧 → 补充"结构整理
4. 自指审计由主 agent 执行，不做递归，只做一轮逐条对照

### 汇总模板

```markdown
## 共识（N个专家均指出）
- 〔D1/D2〕问题 | 严重度

## 分歧
| 问题 | 专家A观点 | 专家B观点 | 判断 |
|------|---------|---------|------|

## 补充（仅1个专家提出但高价值）
- 问题 | 来源 | 严重度
```

## 优化执行技巧

### 安全修改五原则

1. **改前快照**：记录原始内容摘要，供验证对比
2. **改动面与确认计划一致**：计划外的修改列为"附加建议"，不直接执行
3. **保留执行点强化和红线重申**（重复三原则：规则唯一定义点 → references，执行点提醒 → SKILL.md 一句话引用，红线重申 → SKILL.md 禁止句式 + 来源标注）
4. **修改有先后顺序**：先改 references 再改 SKILL.md 中的引用，防孤引用
5. **精简时保留动作指令**：删除冗余内容时，"逐步骤检查"中的"逐步骤"是动作指令，不可精简为"注意"

### 问题排序原则

执行可靠性/幻觉/遵从性 > 输出可执行性/防护 > 结构/字数效率 > 措辞/可读性

### 停止标准

所有"需修复"项已解决、剩余均为"合格"档且用户无进一步要求时，停止优化。把执行点强化当无增量重复删掉，比保留一点冗余更危险。

### 回归验证清单

**结构验证**：
- frontmatter 只含 name + description
- name/目录名/触发语义一致
- SKILL.md 比改前更短更清晰
- 引用文件没有和 SKILL.md 重复大段内容

**语义验证**：
- 问题修复验证：逐条对照计划，每条问题有对应修复
- 改动面一致性：实际修改与计划一一对应
- 触发有效性：优化后 description 仍命中预期场景

**回归验证**：
- Quick-Fail 复检：修改后重新检查 Quick-Fail Flags
- 引用文件可达性：references 中文件都存在且可读

## Quick-Fail 速查表

以下任一为真 → 标记快速失败，优先修复：

| # | Quick-Fail Flag | 检查方法 | 典型案例 |
|---|----------------|---------|---------|
| 1 | description 无法触发目标场景 | 读 description，想象用户会说的话，能否匹配 | description 写"优化"但用户说"改进" |
| 2 | 必须确认的步骤没有卡住 | 遍历工作流，找"确认""等待"类步骤，检查是否有硬编码门槛 | 工作流说"建议确认"但没有"等待用户确认后才进入下一步" |
| 3 | 存在互相矛盾的指令 | 扫描全文，找"必须A"和"不要A"同时出现 | 按需加载条件说"不读"，但流程说"必须检查" |
| 4 | 存在幻影指令 | 扫描全文，找"默默""先验证再输出""精确计算"等不可执行指令 | "输出前先在心中验证格式" |
| 5 | SKILL.md 超过 5000 字（约 300 行）且未使用 references | 统计字数/行数，检查是否有 references 目录 | 8000 字全在 SKILL.md |
| 6 | 引用文件存在大段重复 | 对比 SKILL.md 和每个 reference，找重复段落 | 同一规则在 SKILL.md 和 reference 中完整定义 |
| 7 | Gotchas 完全缺失 | 检查是否有 Gotchas 章节 | 无 Gotchas 或 Gotchas 为空 |
