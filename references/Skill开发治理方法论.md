## 如何使用本文档

**首次使用（约 30 分钟）：**
铁律三条 → 第二节（项目结构）→ 第三节的 1-4 小节 → 第五节（版本控制）→ 第十一节（启动清单）→ 30 分钟上手流程

**遇到问题时查阅：**
铁律一/二 → 第四节（评估体系）→ 第十节（战术手册，按场景索引）
输出质量变差 → 铁律二 + 附录 C（上下文预算）
模型升级后 Skill 失效 → 附录 B（模型版本契约）

**进阶时阅读：**
第七节（分解策略）→ 附录 A（Prompt 工程规范）→ 附录 B/C

**30 分钟上手：完成你的第一个 Skill**

```
1. 发给 Claude Code：
   "请初始化一个新的 Skill 项目，创建 spec/、evals/、src/ 三个目录，
    并初始化版本控制，保存为 v0-baseline。"

2. 用最简单的方式写 spec/AGENT_RULES.md：
   只填 3 样东西：我的 Skill 做什么、给它什么输入、期望什么输出。

3. "请基于 spec/AGENT_RULES.md 生成 evals/test_eval.py，
    只需要 3 个最基本的测试用例。"

   💡 此时运行测试应该全是 FAIL（红灯）——因为还没写代码。
   这恰恰证明测试是有效的。如果没写代码就全绿，说明测试有问题。

4. "请基于 spec/AGENT_RULES.md 在 src/ 下编写 Skill 代码，
    所有修改在隔离分支上进行。"

5. "请运行 evals/test_eval.py，连续运行 3 次确认结果一致。"

6. 3 次都全绿？→ "测试全绿。请保存当前版本并标记为 v1-first-skill。"

7. 没全绿？→ "请修复第一个 FAIL，修完再跑测试。"
```

到这里你已经完成了一个最小可用的 Skill。

## 术语速查

| 术语 | 一句话解释 |
|------|-----------|
| SDD（规格驱动开发） | 先定义什么叫成功，再写代码 |
| EDD（评估驱动开发） | 用测试数据验证成功，而非靠感觉 |
| 断言 / assert | 自动检查规则，类似"质检传感器"：产品过线时滴一声表示合格 |
| JSON Schema | 填表说明——告诉 AI 输出必须有哪些字段、每个字段是什么类型 |
| 回滚 | 撤销，回到之前的版本 |
| 分支 / branch | 平行宇宙，在副本上修改，不影响正式版本 |
| 合并 / merge | 把平行宇宙中确认没问题的修改，带回到正式版本 |
| 标签 / tag | 给某个版本贴个名字标签，方便以后快速找到（如 v1-first-skill） |
| 封版 | 确认当前版本稳定，不再修改，作为正式可用版本 |
| FLAKY | 时红时绿的测试结果，不稳定，需要排查 |
| Red-Green 循环 | 先让测试变红（证明测试有效），再改代码让测试变绿（证明代码有效） |
| Golden Dataset | 你亲手确认过的"标准答案库"——一批"输入→理想输出"的配对 |
| 金标准样本 | Golden Dataset 中最核心的 1-2 个参照，代表"做对了长什么样" |

## 定位

```
SDD（规格驱动开发）    先定义什么叫成功、什么不许动        ← 地基
EDD（评估驱动开发）    用测试数据验证，而非靠感觉          ← 主体
Agentic Workflow       多角色分工，降低单点复杂度（进阶）  ← 上层建筑
```

**SDD 是 EDD 的前提。** 没有规格定义，测试数据就没有筛选标准；没有测试数据，规格就是一纸空文。三层递进，不跳层。

## 一、三条认知铁律

在学习任何具体方法之前，先内化这三条。它们决定了你面对问题时的第一反应是对还是错。

### 铁律一：Skill 是程序，不是文档

发现 Bug 时，第一反应**绝对不是**在 Prompt 里加一条"请注意XXX"。

**正确反应：** 写一个能稳定复现这个 Bug 的测试用例，加入 evals/test_eval.py。Prompt 里的规则，上下文一长就被 AI 遗忘；写进测试脚本的断言，永远不会失效。

> **没有执行证明的规则，等于不存在。** Prompt 里声明的"红线"或"禁令"，必须落地为硬评估的断言或软评估的打分指标，否则它只是一厢情愿。

### 铁律二：规则模块化，按需加载

输出质量下降时，本能反应是加规则。但往往问题出在**旧规则互相打架**——一条说"要简洁"，另一条说"要详细"，AI 两边都想遵守，结果两边都做不好。

**先做减法或拆分，不做加法：**
- 输出变差了，先检查现有规则有没有互相矛盾的
- 如果矛盾规则中有一条是冗余的 → 直接删
- 如果两条都有业务价值但互相打架 → 拆分到不同模块，按需加载

**规则多了不一定要砍——要拆：**

```
spec/
├── AGENT_RULES.md        ← 核心规则（始终加载）
├── rules_parse.md        ← 解析规则（仅解析任务时加载）
├── rules_tone.md         ← 语气规则（仅润色任务时加载）
└── rules_edge.md         ← 边缘 case 规则（仅排查时加载）
```

**什么时候启动拆分（出现任一条就拆）：**
- 两条规则在同一个任务中给出矛盾指令
- 一条规则只在特定阶段有用，但被所有阶段加载
- 排查后发现质量下降不是规则缺失，是规则太多导致 AI 无法判断优先级

**不设条数硬上限。** 10 条规则如果清晰无冲突，不需要拆；5 条规则如果有冲突，也该拆。以清晰无冲突为准。

**关键区分：删规则，不删示例。** 规则是"你应该怎么做"的指令，示例是"做对了长什么样"的参考。规则多了会打架，示例多了反而帮 AI 理解你要什么。

**给 Claude Code 的排查指令：**
```
"当前输出质量下降了。
在新增任何规则之前，请先检查 spec/ 下的所有规则文件中
是否有相互矛盾或重叠的规则。
列出你发现的矛盾，建议删除或拆分到独立模块。
注意：不要删除任何金标准样本或示例，只审查指令性规则。
如果存在职责混杂（某些规则只在特定阶段有用），
建议按职责拆分为 rules_[职责].md 模块。"
```

**规则模块动态加载指令：**
```
"当前是 [解析/润色/边缘排查] 阶段，请加载以下规则文件：
1. spec/AGENT_RULES.md（核心规则，始终加载）
2. spec/rules_[当前阶段].md（阶段规则）
不需要加载其他阶段的规则模块。"
```

### 铁律三：你的判断力在业务层，不在代码层

输入一个东西，出来的对不对，你最清楚。代码写得优不优雅，不是你的事。

你定义成功标准（业务语言）→ AI 生成代码 → 评估工具验证 → 你做最终取舍。你永远不需要看懂代码。

## 二、项目文件结构

```
项目根目录/
├── spec/                     ← 契约层：定义"什么叫成功"
│   ├── AGENT_RULES.md        ← 成功标准、绝对红线、容忍清单、版本控制规范
│   ├── eval_rubric.md        ← 软评估量表 + 金标准样本
│   └── api_schema.json       ← 输出结构的机器契约（结构化输出时推荐）
│
├── evals/                    ← 评估层：验证"是否真的成功"
│   ├── test_eval.py          ← 硬评估脚本（PASS/FAIL，不过不能发布）
│   ├── soft_eval.py          ← 软评估自动化裁判（进阶，见第四节）
│   └── golden_dataset/       ← Golden Dataset（输入输出 Pair）
│       ├── case_01_input.md
│       └── case_01_expected.md
│
├── src/                      ← 执行层：AI 生成的业务代码
│   └── (skill 的实际代码)
│
└── .git/                     ← 版本控制
```

**三段式命名逻辑：**
- **`/spec`** — 不是说明书，是 AI 必须执行的契约
- **`/evals`** — 统管硬评估（Python 断言）和软评估（LLM 打分）
- **`/src`** — 代码和契约物理隔离，防止 AI 修代码时顺手改了评估脚本

**关于 api_schema.json：**
你的 Skill 输出能用一个固定的数据结构描述吗？能 → 用 api_schema.json，test_eval.py 直接校验输出是否符合 Schema，比手写断言更精确；不能（自由文本）→ 不用。

**给 Claude Code 的初始化指令：**
```
"请按以下结构创建项目目录：
- spec/ 放 AGENT_RULES.md 和 eval_rubric.md
- evals/ 放 test_eval.py 和 golden_dataset/
- src/ 放所有业务代码

同一个版本记录中严禁同时包含 src/ 和 spec/ 或 evals/ 的修改。
结构性变更时，先保存 evals/ 的修改（确认红灯），
再保存 src/ 的修改（确认绿灯），分两次保存。"
```

## 三、spec/AGENT_RULES.md — 规格文件

### 1. 成功基准线（Acceptance Criteria — 怎样算"做对了"）

用业务语言写，只关注"给它什么、出来什么"，不管代码内部怎么实现：

```
【Skill 名称】：_______________
【核心功能】：_______________（一句话）
【输入】：_______________（举一个具体例子）
【期望输出】：_______________（格式 + 内容）
【判定标准】：_______________（怎样算"对了"）
```

### 2. 绝对红线（Hard Constraints）

一旦触发，立即回滚，不继续修补：

- **业务中断** — 崩溃、死循环、跳过核心步骤
- **数据安全** — 覆盖/删除原始数据、数据截断或丢失
- **过度工程** — 未经允许引入新依赖、重构不相关的模块、"顺手"改了别的东西

### 3. 容忍清单（Ignored Warnings — 明确告诉 AI"这些问题不许你修"）

以下问题，AI 修了反而可能引入新 bug，一律不许修：

- □ 代码里有黄色警告线（不影响运行，只是"不够优雅"）
- □ AI 建议"可以用更现代的写法"（现在能跑就行）
- □ 极少出现的奇怪输入导致的小问题（不影响正常使用）
- □ AI 扫描出的"建议性"优化（不修也不影响功能）

看到 AI 建议修这些问题时，回复：*"这个问题在容忍清单里，不需要修。请只关注我要求的修改。"*

### 4. 输出品质要求（Quality Spec）

定义无法用 Python 断言的质量维度（仅当输出含自然语言时需要）：

```
【语气要求】：_______________
【禁止出现】：_______________
【金标准样本】：（贴 1-2 个你认为"对的"输出）
```

> **术语说明：** "金标准样本"指单个"做对了长什么样"的参考（1-2 个）；"Golden Dataset"指你确认过的完整测试用例库（3-50+ 个）。金标准样本是 Golden Dataset 中最核心的参照。

### 5. 铁规：契约文件必须同步（按变更类型分流）

spec/AGENT_RULES.md、evals/test_eval.py、spec/eval_rubric.md 必须同步（如果项目使用了 api_schema.json，也包含在内）。最大风险不是 bug 修不好，而是**测试本身是错的——全绿但其实没在检测你要的东西**。

但并非所有变更都需要走同一套流程。**按变更性质分流：**

**结构性变更 → 强制 Red-Green 循环：**

增删字段、改步骤、调格式等"骨架"级修改：

```
## 契约文件同步规则 — 结构性变更
- 先修改 evals/test_eval.py，确认新测试在当前代码上会 FAIL（红灯）
- 然后再修改 src/ 下的代码，让测试变 PASS（绿灯）
- 测试改动和代码改动分两次保存版本
- 顺序永远是：改测试 → 确认红灯 → 改代码 → 确认绿灯
```

**语义性变更 → 软评估直裁：**

调语气、改人味、调排版等"血肉"级修改，强写 Python 断言是效率浪费：

```
## 契约文件同步规则 — 语义性变更
- 不需要先造红灯
- 修改后直接运行 Skill，将输出交给软评估打分
- 软评估所有维度均 ≥ 3 分方可合并
- 仍需在隔离分支上进行
- 调整后必须先重跑硬评估确认全绿，再进入软评估
```

**如何判断属于哪类？**

| 你要改的是…… | 分类 |
|-------------|------|
| 增加/删除一个输出字段 | 结构性 |
| 修改处理步骤的顺序 | 结构性 |
| 换输出格式（如 JSON→表格） | 结构性 |
| 改语气（如正式→亲切） | 语义性 |
| 改措辞（如"禁止"→"请勿"） | 语义性 |
| 拿不准？ | 先按语义性做，更简单 |

**给 Claude Code 的指令：**
```
"我要修改 [某功能]。先判断这是结构性变更还是语义性变更：

如果是结构性变更（字段、步骤、格式）：
  先修改 evals/test_eval.py 使当前代码 FAIL，
  确认 FAIL 后再修改主代码让它 PASS。分两次保存。

如果是语义性变更（语气、人味、排版）：
  直接修改主代码，修完后将输出交给软评估打分。"
```

> **Red-Green 循环（先红后绿）适用于结构性变更：** 先让测试变红（FAIL）——证明测试确实在检测这个功能；再改代码让测试变绿（PASS）——证明功能确实修好了。**红灯是测试的测试**——跳过红灯，绿灯可能是假的。语义性变更由软评估做最终裁决。

## 四、三层评估体系

```
第一层：硬评估（能不能跑通）      — test_eval.py     不过不能发布
第二层：软评估（质量好不好）      — eval_rubric.md    打分参考，人做最终判断
第三层：性能评估（跑得快不快）    — 静态预判 + 运行时断言 + 规模压测
封版条件：硬评估稳定全绿 + 软评估所有维度 ≥ 3 分 + 性能评估通过 + 人工审查
```

### 第一层：硬评估（test_eval.py）— 不过就不能发布

硬评估 = 用 Python 脚本自动检查，结果只有 PASS 或 FAIL，没有灰色地带。

**做什么：** 只检查"骨架"——格式对不对、字段全不全、步骤有没有跳、有没有崩溃。
**不做什么：** 不检查"血肉"——文字好不好、语气对不对、有没有人味。

> ⚠️ 一旦把"不能出现某个词"之类的语言规则写进 test_eval.py，就会越维护越崩——同一个意思有一百种说法，用代码去卡语言问题是死路。语言问题交给软评估。

```
python evals/test_eval.py
→ Case 1: PASS
→ Case 2: FAIL (原因：缺少字段X)
→ Case 3: FLAKY (3次中2次PASS，需排查)
```

**封版标准：稳定全绿，而非一次全绿。** AI 输出是概率性的——同样的输入跑两次，输出可能不同。封版前对 test_eval.py **连续运行 3 次**：
- 3 次结果完全一致 → 可以封版
- 出现不一致 → 标记为 FLAKY 用例，进入排查
- 不允许用"多跑几次总有一次过"来封版

**测试可信度自检（首次测试后必做）：**
```
发给 Claude Code：
"请检查 evals/test_eval.py 中每个测试用例：
1. 每个断言是否真的在验证 AGENT_RULES.md 中的要求？
2. 如果把输出改成完全错误的值，这个断言会 FAIL 吗？
   （如果不会，说明断言无效）
3. 有没有测试用例只有注释没有断言？"
```

**FLAKY 用例处理：** 时红时绿的用例不参与 PASS/FAIL 判定。尝试加强 Prompt 约束；仍然 FLAKY → 改由软评估覆盖。

### 第二层：软评估（eval_rubric.md）— 打分参考，人做最终判断

软评估 = 让另一个 AI 对照评分表打分（1-5 分），帮你判断代码判不了的东西。**分数是参考，不是命令——最终你说了算。**

以下是**常见维度示例**，每个项目选取 2-4 个最相关的，不要照搬：

| 维度（示例） | 5分 | 3分（及格线） | 1分 |
|---|---|---|---|
| 人味 | 完全像人写的 | 能用，偶有AI痕迹 | 纯AI输出 |
| 语气 | 完全符合要求 | 基本对路 | 明显不对 |
| 连贯性 | 有清晰推理链 | 段落间有逻辑 | 跳跃、断裂 |
| 冗余度 | 精练，无废话 | 略有冗余 | 大量套话 |

**选择原则：** 说不出为什么需要某个维度，就不要加。数据同步类可能完全不需要软评估，内容生成类可能需要 3-4 个维度。

**软评估的三种执行方式（推荐度递减）：**

**方式一：自动化裁判（推荐，日常迭代用）**

让 Claude Code 生成一个评估脚本 evals/soft_eval.py，自动调用轻量级模型打分：
```
"请编写 evals/soft_eval.py，挂载在 test_eval.py 之后运行。要求：
1. 读取 golden_dataset/ 中的输入，用当前 Skill 生成输出
2. 将输出 + eval_rubric.md + 金标准样本发送给轻量级模型 API
3. 按 eval_rubric.md 的维度打 1-5 分
4. 所有维度均 ≥ 3 分则整体 PASS
5. API key 从环境变量读取，不要硬编码"
```

如果设置 API key 遇到困难，**先用方式二手动评估完全没问题**。

**方式二：轻量快捷版（快速检查用）**

在另一个 Claude 对话中：
```
"请用一句话评价以下输出的质量（1-5分），
重点看有没有人味、有没有废话。
只给分数和一句话理由。

[贴输出]"
```

**方式三：完整手动评估（封版前用）**

在另一个 Claude 对话中，贴实际输出 + eval_rubric.md + 金标准样本，让它打分。所有维度均 ≥ 3 分可封版。

**关键：不要用写代码的同一个 Claude Code 会话做软评估。运动员不能当裁判。**

**防废弃策略：** 日常迭代用方式一（零手动成本）。没搭自动化时用方式二过渡。方式三只在封版前做一次。

### 第三层：性能评估（跑得快不快）

硬评估管"能不能跑通"，软评估管"质量好不好"，性能评估管"跑得快不快"。10 行数据秒过、10 万行跑 10 分钟的情况，前两层完全感知不到。

**三层递进，不需要每次全做：**

**① 静态预判（绿线阶段就开始做）：**

```
发给 Claude Code：
"请分析当前 Skill 的性能预判指标：
1. 统计当前加载的所有规则文件的总 token 数（估算）
2. 列出流程中是否有重复调用的节点
3. 检查是否有中间结果被重复生成
4. 当前阶段是否加载了不需要的规则模块
5. Few-shot 示例数量是否超过必要限度
如果任何一项有问题，给出具体的精简建议。"
```

**② 运行时断言（黄线阶段加入）：**

在 test_eval.py 中增加性能基线——首次封版时记录实际耗时，后续封版检查是否退化：

```
发给 Claude Code：
"请在 evals/test_eval.py 中增加一个性能测试用例：
记录当前 Skill 处理标准测试输入的耗时作为基线，
后续运行时如果耗时超过基线的 1.5 倍，标记为 FAIL。"
```

**③ 规模压测（蓝线封版前做）：**

```
发给 Claude Code：
"请用以下三种数据规模分别运行 Skill，记录每次的耗时：
1. 10 条数据（基线）
2. 100 条数据
3. 1000 条数据
判断耗时增长是线性的还是超线性的。
如果是超线性增长，分析瓶颈在哪里。"
```

**什么时候做：** 绿线只做①；黄线加入②；蓝线封版前做③。

## 五、版本控制安全网（金丝雀发布）

> 金丝雀发布（Canary Release）：矿工下井前放一只金丝雀试探毒气——AI 每次改代码也一样，先在隔离的安全分支上试，确认没问题再合并到正式版本。
> **绝对禁止在正式版本（main）上直接修 bug。**

> 每个操作给出两层——**自然语言指令**（发给 Claude Code）+ **底层命令**（供你了解原理）。日常用自然语言指令即可。

**项目初始化（只做一次）：**
```
发给 Claude Code：
"请初始化项目的版本控制，把当前所有文件保存为初始版本，
标记为 v0-baseline。"

// 底层命令：git init && git add -A && git commit -m "init" && git tag v0-baseline
```

**每次修 bug 前：**
```
"请创建一个隔离的工作分支，命名为 fix/[问题简述]。
在这个分支上修复，修完后运行 evals/test_eval.py。"

// 底层命令：git checkout -b fix/[问题简述]
```

**修完后的三种情况：**

```
情况 A：test_eval.py 全 PASS（连续 3 次一致）
→ "测试全部通过。请将这个分支合并到主线版本，
   打上版本标签 v[N]-[描述]，然后删除工作分支。"
  // 底层：git checkout main && git merge fix/[问题] && git tag v[N]-[描述]

情况 B：有 FAIL，但没引入新错误
→ "继续在当前分支修复第一个 FAIL，不要动其他东西。"
  ⚠️ 同一个 FAIL 连续修复 3 次仍未 PASS → 按情况 C 处理

情况 C：修复导致 2 个以上新的 FAIL
→ 先排除非确定性（对失败用例单独跑 3 次，3 次都 FAIL 才确认）。
  确认后立即止损：
  "这次修复引入了新问题，放弃这个分支。
   回退到主线版本。
   创建一个新分支，用完全不同的技术路径重新解决原问题。"
  // 底层：git checkout main && git branch -D fix/[问题] && git checkout -b fix/[问题]-v2

  回退后，先让 AI 说思路再动手：
  "回退到主线后，请分析上一次修复失败的原因，
   告诉我你打算用什么不同方法解决。不要开始写代码。"
```

### 为什么情况 C 要强制换路径？

AI 在烂摊子上继续修补，只会越补越烂——它会一直沿着错误的方向打补丁。强制回到干净版本、换一种完全不同的方法重做，反而比死磕更快。

### 给 Claude Code 的全局指令（写入 spec/AGENT_RULES.md）

```
## 版本控制工作规范
- 所有修改必须在隔离分支上进行，严禁直接修改主线版本
- 每次保存版本前必须运行 evals/test_eval.py
- 如果修复导致超过 2 个新的 FAIL，立即停止并报告
- 严禁在失败的分支上继续叠加修复
- 同一个 FAIL 连续修复 3 次仍未 PASS，放弃当前分支，换路径
```

### 版本控制出错时的降级方案

```
发给 Claude Code：
"如果版本控制操作失败（如合并冲突、状态异常），
不要尝试自动解决，请立即：
1. 列出当前文件状态
2. 告诉我当前处于什么状态
3. 等待我的指示"
```

**如果实在搞不定，终极后退方案：**
```
"停止所有操作。
请把当前项目目录下所有文件打包成 backup_[日期].zip，
放到项目根目录。
然后我们从头开始排查。"
```

## 六、Golden Dataset（黄金数据集）— 你确认过的"标准答案库"

### 什么是 Golden Dataset

你亲手确认过的"输入→理想输出"配对，代表真实业务场景。不是 AI 生成的，是你自己看过、认可的。

```
evals/golden_dataset/
├── case_01_input.md       ← 真实业务输入
├── case_01_expected.md    ← 你确认过的理想输出
└── ...
```

### 规模建议

| 阶段 | 数量 | 用途 | 毕业条件 |
|---|---|---|---|
| 绿线 | 3-5 个 | 验证核心功能 | 硬评估稳定全绿 |
| 黄线 | 10-20 个 | 覆盖主要变体 | 软评估所有维度 ≥ 3 分 |
| 蓝线 | 20+ 个 | 回归测试 + 对抗样本 | 包含 ≥ 5 个对抗样本，全 PASS |

### 如何积累

**绿线阶段：** 自然生长。跑通一次，好的输出存下来。

```
"请把刚才的输入保存为 evals/golden_dataset/case_0X_input.md，
 输出保存为 evals/golden_dataset/case_0X_expected.md。"
```

**黄线阶段：** 不能只靠自然积累，必须主动设计对抗样本：

```
"请基于 AGENT_RULES.md 的成功基准线，
为 evals/ 设计 5 个对抗性测试用例：
1 个空输入、1 个超长输入、1 个格式错误、
1 个含特殊字符、1 个语言混杂。
每个用例只要求不崩溃、不丢数据，不要求正确输出。"
```

### 与 test_eval.py 的关系

test_eval.py 从 Golden Dataset 中抽取用例。Dataset 扩大了，test_eval.py 跟着扩展。Dataset 是源头，test_eval.py 是它的可执行形式。

## 七、复杂 Skill 的分解策略

### 判断标准（出现任一条就拆）

- test_eval.py 需要验证三种以上完全不同的能力
- 修一个功能经常破坏另一个功能
- AGENT_RULES.md 的成功基准线超过三条

### 拆分方式

不需要搭多智能体架构。**把一个大 Skill 拆成几个小 Skill，串行调用：**

```
之前：skill_monolith/          一个 skill 做所有事

之后：
skill_parse/      ← 只负责解析输入
skill_transform/  ← 只负责数据转换
skill_render/     ← 只负责生成输出（"人味"在这里）
```

每个小 Skill 有自己的 spec/AGENT_RULES.md 和 evals/test_eval.py。

### 拆分后的管线集成测试

每个子 Skill 单独绿 ≠ 管线全绿。拆分后必须增加端到端验证：

```
"请用第一步的真实输出喂给第二步，验证端到端管线不中断。"
```

### 进阶：多角色协作（Agentic Workflow）

当拆分后的小 Skill 都调稳了，再考虑用 Agent 编排：

```
Router Agent   → 判断意图，分发任务
Worker Agent   → 执行硬核逻辑
Judge Agent    → 对 Worker 输出打分，不过就自动重试（最多 3 次）
Polisher Agent → 注入"人味"，润色最终输出
```

**跃迁信号：** 子 Skill 之间的衔接逻辑复杂到需要手动编排调用顺序时，引入 Router Agent。

**这是毕业阶段的架构，不是入门阶段的要求。**

## 八、分层推进策略

```
⚫ 安全层    版本控制初始化、测试环境就绪、绝对红线生效
  ↓
🟢 绿线     硬评估稳定全绿（连续 3 次一致）
  ↓
🟡 黄线     软评估所有维度均 ≥ 3 分
  ↓
🔵 蓝线     Golden Dataset ≥ 20 且包含 ≥ 5 个对抗样本，全 PASS
  ↓
🟣 紫线     复杂 Skill 拆分完成，子 Skill 各自达到黄线
  ↓
🔴 极线     平台限制 workaround、极端边缘 case
```

**每一层稳了才进下一层。**

### 各层级毕业条件

| 层级 | 毕业条件 |
|------|---------|
| ⚫→🟢 | 硬评估稳定全绿（连续 3 次一致） |
| 🟢→🟡 | 软评估所有维度均 ≥ 3 分 |
| 🟡→🔵 | Golden Dataset ≥ 20，包含 ≥ 5 个对抗样本，全 PASS |
| 🔵→🟣 | 触发拆分条件时拆分完成，子 Skill 各自达到黄线 |
| 🟣→🔴 | 主流程 + 子 Skill 连续 5 次修改无新 FAIL |

> **极线 ≠ 绝对红线。** 极线是最高进阶阶段（处理平台限制和极端边缘 case），绝对红线是不可触碰的禁令（数据丢失、业务中断等）。两者完全不同。

### 成熟度梯度：控制随信心递减

```
初期（不信任期）    → 重度管控：每次只改一个文件，注释骨架，拒绝重构
中期（建立信心期）  → 适度放权：允许 AI 在单个模块内自由实现，保持评估验收
成熟期（信任期）    → 轻量治理：主要靠 Golden Dataset 回归测试，规则精简到核心几条
```

**判断标准：** 连续 5 次修改都没触发新 FAIL → 考虑松开一层管控。管控的目的是建立信任，不是永远锁死。

## 九、工作流速查

### 每次调试

```
1. 创建隔离分支 fix/[问题]
2. 修复第一个 FAIL（只修一个）
3. 运行 evals/test_eval.py（连续 3 次确认稳定）
4. 全 PASS → 合并到主线版本，打标签
5. 有 FAIL 但没有新增 → 继续修（同一 FAIL 超 3 次 → 换路径）
6. 新增 ≥ 2 个 FAIL → 排除非确定性后确认，放弃分支，换路径
7. 硬评估稳定全绿 → 做软评估
8. 软评估所有维度均 ≥ 3 分 + 人工审查 → 封版
```

### 给 Claude Code 的指令模板

**调试时：**
```
请基于 spec/AGENT_RULES.md 测试当前代码。
1. evals/test_eval.py 全 PASS 且没触发绝对红线
   → 回复"契约达成，可以封版"，严禁提优化建议
2. 只有触发绝对红线时才能报告错误，用最少代码修复，严禁重构
3. 所有修改在隔离分支上进行，只改 src/ 下的文件
4. 如果修复导致超过 2 个新 FAIL，立即停止并报告
5. 同一个 FAIL 连续修复 3 次仍未 PASS，停止并报告
```

**开发新 Skill 时：**
```
请先基于 spec/AGENT_RULES.md 生成：
1. evals/test_eval.py（5 个用例，PASS/FAIL）
2. spec/eval_rubric.md（评分量表 + 金标准样本）
两个文件经我确认后，才能在 src/ 下写主干代码。
所有开发在隔离分支上进行。
```

## 十、实操战术手册

18 条战术按**工作场景**分为 4 组。遇到问题时先想"我现在处于什么场景"，直接翻对应分组。

```
🟢 每次动手前    养成的固定习惯，不管做什么都先做这几件
🔒 管住 AI      AI 不听话、想乱来时的应对
🛠 出了问题      卡住、改崩、越修越乱时怎么办
✅ 确认与封版    改完后确认结果、保存版本
```

---

### 🟢 每次动手前（养成习惯，每次都做）

#### 改动前快照（Pre-change Snapshot）

```
"在修改任何代码之前，请先保存当前状态的快照，
版本说明为 'snapshot before fix: [问题描述]'。
确认快照保存成功后再开始修改。"
// 底层：git add -A && git commit -m "snapshot before fix: [问题描述]"
```

#### 限定修改范围（Scope Lock）

```
本次修复只允许修改以下文件：
- [文件名]
严禁修改任何其他文件。需要改其他文件请先告诉我原因。
```

#### 一个会话只干一件事（Session Discipline）

| 会话类型 | 目的 | 结束条件 |
|---|---|---|
| 开发会话 | 写新代码 | test_eval.py 全绿 |
| 修复会话 | 修一个 FAIL | 该 FAIL 变 PASS 且无新 FAIL |
| 审查会话 | 检查变更内容 | 确认改动范围合理 |
| 评估会话 | 软评估打分 | 拿到分数和改进建议 |

**绝对不要在一个会话里又开发、又修 bug、又做评估。**

#### 一次只修一个问题（Single Variable Debug）

```
"当前有多个问题，但你每次只能修一个。
请告诉我你认为最关键的那一个，
修完后立即运行 evals/test_eval.py。
在我确认结果之前，不要开始修第二个。"
```

---

### 🔒 管住 AI（AI 不听话时的应对）

#### 拒绝重构（Anti-Refactor Shield）

```
"不要重构。当前结构即使不优雅也是可以工作的。
在不改变现有结构的前提下修复这个 bug。
如果你认为不重构就无法修复，请解释原因，
但在我同意之前不要动任何结构。"
```

**阶段适配：** 绿线/黄线阶段严格执行。蓝线及以上，如果 AI 提出的重构能通过全部 Golden Dataset 回归测试，可以允许，但仍需在隔离分支上进行。

#### 禁止加依赖（Dependency Lock）

**写入 spec/AGENT_RULES.md：**
```
## 依赖管理
- 严禁未经允许引入新的 pip/npm 依赖
- 必须引入新库时，先说明：这个库是什么、为什么现有依赖解决不了、有没有替代方案
- 等待我明确同意后才能安装
```

#### 注释骨架锁定（Comment Skeleton Lock-in）

**触发条件：** AI 每次修 bug 都偷偷重排你的业务逻辑顺序。

你自己在文件里写好中文注释骨架：
```python
# ===== 第一步：读取本地文件 =====
# ===== 第二步：提取关键字段 =====
# ===== 第三步：转换为目标格式 =====
# ===== 第四步：写入目标系统 =====
```

然后发：
```
"我已经写好了四步中文注释骨架。
只在每个注释的下方填写实现代码。
绝对红线：严禁合并步骤、严禁修改或删除注释、
严禁改变执行顺序、每一步不超过 15 行。"
```

**为什么有效：** 你不需要懂代码，但你懂业务流程。用自然语言锁定骨架，AI 就失去了"重构"的空间。

#### 反向逼问（Reverse Interrogation）

**触发条件：** AI 输出一大段分析，列了 4 个风险，提出重构方案。

```
"先不讨论你提出的这些风险。
只用'会'或'不会'回答我一个问题：

如果我完全无视你说的这 4 个风险，
这段代码在处理一个标准的正常输入时，
主流程会中断吗？

回答'不会' → 立刻停止讨论这些风险，结束本次审查。
回答'会' → 只告诉我哪一个风险会导致主流程中断，其余不讨论。"
```

---

### 🛠 出了问题（卡住、改崩、越修越乱）

#### 切断上下文污染（Context Reset）

**触发条件：** 同一个对话里来回修改超过 3 轮，越修越乱。

```
1. 立即结束当前会话
2. 开启全新对话
3. 只把引发问题的那一个文件 + 报错信息发给新对话
4. 不要发整个项目，不要发历史讨论
5. 拿到修复代码后，自己手动替换进去
```

如果不确定该贴什么：
```
"请把与这个报错相关的代码和完整报错信息整理出来，方便我复制到新对话。"
```

**给新对话的指令：**
```
我有一个 Python 脚本出了问题。
请只针对这个报错给出修复代码，
不要修改任何其他部分，不要重构，不要优化。

【代码片段】：[贴报错附近的代码]
【报错信息】：[贴完整 error traceback]
```

#### 紧急恢复（Emergency Recovery）

```
"停止所有操作。
1. 列出最近的 10 个版本记录及其内容摘要
2. 暂存当前未保存的修改
3. 我来决定回滚到哪个版本"
// 底层：git log --oneline -10 && git stash
```

**连版本控制都没初始化时：**
```
"停止所有操作。
把当前所有文件打包成 backup_[日期].zip，放到项目根目录。
然后从头开始排查。"
```

#### 硬编码短路（Hardcoded Bypass）

**触发条件：** 非核心模块报错导致整个主流程卡死。

```
"这个模块的 Bug 先不修。
请将这个函数的复杂逻辑注释掉，
直接强制返回写死的假数据：{'status': 'ok', 'data': '测试占位'}
先让下游跑通，确认没问题后再回来修。"
```

**⚠️ 必须同时做两件事：**
1. 在代码里留 `# TODO: 这里是假数据，必须替换`
2. 在 test_eval.py 中写一个对应的跳过用例，防止忘记移除假数据

#### 暴力异常兜底（Try-Except Shielding）

```
"不要写复杂的条件判断来处理每种异常。
直接在这段代码外层套一个全局错误捕获，
出错时将跳过的条目记录到 skipped_items.log，
在终端打印'⚠️ 跳过该条目: [报错原因]'，
然后强制继续执行下一条。绝不允许程序中断退出。"
```

**什么时候用：** 绿线阶段，只需主流程跑通。
**什么时候不用：** 涉及数据写入的操作——写操作出错被"吃掉"可能造成脏数据。

#### 差异锚定（Diff-Only Prompting）

**触发条件：** V1 能跑，AI 改成 V2 后崩了。

**在新对话中发：**
```
"这是能跑通的 V1 版本代码：[贴相关片段]
这是崩溃的 V2 版本：[贴同一片段]
报错信息：[贴报错]

你的任务：
1. 只对比 V1 和 V2 的差异
2. 告诉我是 V2 的哪一行引发了报错
3. 只给出那一两行的修复代码
4. 严禁输出完整文件，严禁重构"
```

#### 规则冲突调试（Rule Conflict Debug）

**触发条件：** 加了几轮规则后质量反而下降。

```
"当前输出质量下降了，我怀疑是规则冲突导致的。

1. 逐条审查 spec/ 下所有规则文件
2. 找出相互矛盾或重叠的规则
3. 列出矛盾对
4. 建议删除、保留、或拆分到独立模块
5. 在我确认前不要动任何文件"
```

**进阶：规则模块化拆分**
```
"请将所有规则分类为：
- 核心规则：删了主流程就崩
- 阶段规则：只在特定阶段有用
- 可疑规则：和其他规则有冲突

核心规则保留在 AGENT_RULES.md，
阶段规则按职责拆到 rules_[职责].md 模块。"
```

**什么时候用：** 每次你觉得"越调越差"的时候，先跑这个。90% 的情况下，删规则或拆分比加规则有效。

#### 变更审计（Diff Audit）

```
"请展示从上一次保存到现在的所有修改，
用中文简要说明每处改动的目的。
不要说'优化了代码结构'这种废话，
要说'把第 35 行的路径从 /tmp 改成了 /data，因为原路径不存在'。"

```

**进阶——让 AI 自证清白：**
```
"请列出你刚才的所有改动，逐条回答：
1. 改了什么？ 2. 为什么改？
3. 是否在我授权范围内？ 4. 有没有可能影响其他功能？"
```

#### 输出稳定性验证（Output Stability Verification）

```
"请用相同的测试输入运行 Skill 3 次，
对比 3 次输出的核心结构是否一致（字段、步骤、关键结论），
忽略措辞差异。3 次结构一致但措辞不同，视为稳定。
同时展示修改前后的关键差异。只展示结果，不解释代码。"
```

#### 异步软评估（Separate Judge Session）

**优先：** 自动化裁判 → `"请运行 evals/soft_eval.py"`

**手动（搭建前过渡）：** 打开 claude.ai 新对话，贴输出 + eval_rubric.md + 金标准样本，让它打分。

**为什么必须新对话或独立脚本：** 运动员不能当裁判。

#### 锁定健康版本（Version Pinning）

```
"当前版本测试全部通过。请保存当前所有文件，
提交版本说明为 'stable: [功能描述] - all tests pass'，
打上版本标签 v[N]-stable。
请告诉我标签名称，以便回滚。"
// 底层：git add -A && git commit && git tag v[N]-stable
```

**回滚：**
```
"当前修改导致了新问题，请回滚到 v[N]-stable，
并确认 test_eval.py 全绿。"
// 底层：git checkout v[N]-stable
```

## 关键认知

**三条铁律（最重要）：**
1. **Skill 是程序，不是文档。** 发现 bug 第一反应是写测试用例，不是改 prompt。没有执行证明的规则等于不存在。
2. **规则模块化，按需加载。** 质量下降时先删矛盾规则或拆分到独立模块，不做加法。不设条数硬上限，以清晰无冲突为准。
3. **你的判断力在业务层。** 规格你来定，评估数据你来确认，代码让 AI 写。你永远不需要看懂代码。

**AI-Native 要点：**
4. **AI 输出是概率性的。** 封版标准是"稳定全绿"（连续 3 次一致），不是"一次全绿"。FLAKY 用例不参与判定。
5. **先跑通，再质量，最后效率。** 硬评估管骨架（能不能跑），软评估管血肉（质量好不好），性能评估管速度（跑得快不快）。
6. **性能不是事后优化，是三层递进。** 静态预判 → 运行时断言 → 规模压测。绿线只做预判，黄线加断言，蓝线封版前压测。
7. **Red-Green 循环只用于结构性变更；语义性变更由软评估直裁。** 改字段、改步骤 → 先红后绿；改语气、改人味 → 软评估打分。Prompt 变更视为结构性变更；多 Prompt 模块使用时必须做组合测试（见附录 A）。
8. **软评估自动化优先。** 日常用自动化裁判，封版前做完整评估，避免手工搬运导致评估被废弃。

**方法论要点：**
9. **SDD 是地基，EDD 是主体，Agentic Workflow 是上层建筑。** 三者递进，不跳层。
10. **契约文件必须同步。** 结构性变更分两次保存；语义性变更先硬评估确认再软评估打分。
11. **金丝雀发布：AI 永远在隔离分支上工作。** 新增 ≥ 2 个 FAIL 就放弃分支、换路径。同一 FAIL 超 3 次也换。
12. **Golden Dataset 是生长出来的。** 绿线阶段自然积累，黄线阶段主动设计对抗样本。
13. **复杂 Skill 调不稳，先拆分再调试。** 一个 Skill 只做一件事，拆分后做端到端管线测试。
14. **容忍清单防止 AI 擅自优化不该修的东西。** 每个项目都要明确"什么不修"。
15. **成熟度梯度。** 初期重度管控 → 中期适度放权 → 成熟期轻量治理。管控的目的是建立信任，不是永远锁死。

## 附录 A：Prompt 工程规范（进阶）

当项目成熟后，业务规则和 Prompt 指令应分离管理：

```
spec/
├── AGENT_RULES.md        ← 业务规则（人写的，定义成功标准）
├── prompts/              ← Prompt 库（AI 读的，驱动行为）
│   ├── system.md         ← 系统 Prompt
│   ├── rules_parse.md    ← 解析阶段 Prompt
│   └── CHANGELOG.md      ← Prompt 变更日志
```

初期不需要拆分。当触发铁律二的拆分信号时，连同 Prompt 库一起拆。

**Prompt 变更日志（CHANGELOG.md）：** 每次修改 Prompt，记录改了什么、为什么改、评估结果。

**Prompt 变更走 Red-Green 循环：** 对 Prompt 的修改视为结构性变更。

**Prompt 组合测试（使用多个 Prompt 模块时必做）：**

当 Skill 使用多个 Prompt 模块时，必须测试组合效果——单个模块没问题不代表组合后没问题：

```
"请在以下三种配置下分别运行 Skill 并对比输出：
1. 只加载 rules_parse.md
2. 只加载 rules_tone.md
3. 同时加载 rules_parse.md + rules_tone.md
对比 3 次输出，检查组合后是否出现指令冲突。"
```

## 附录 B：模型版本契约（进阶）

模型升级后 Skill 可能全面失效。在 spec/AGENT_RULES.md 中声明：

```
## 模型契约
- 目标模型：[模型名称及版本]
- temperature：0（确定性优先）
- 最后一次全量验证日期：[日期]
```

**模型升级时：** 冻结所有代码和 Prompt，只换模型，跑全量 test_eval.py。全绿 → 更新版本号封版。有红 → 最小适配，不重写。

**规则的模型相关性标注：**
```
- "输出 JSON 格式" → [模型无关] 所有主流模型都能遵从
- "不要使用列表格式" → [模型相关] 部分模型会忽略否定指令
- "用同事口吻" → [模型相关] 高度依赖模型对语气的理解
```

模型升级时优先重测模型相关规则。

## 附录 C：上下文预算规范（进阶）

规则模块化解决了冲突问题，但没有解决容量问题。

**上下文预算分配（在 spec/AGENT_RULES.md 顶部声明）：**

```
## 上下文预算（模型：Claude Sonnet，窗口：200K tokens）
| 分区 | 预算 | 实际 |
|------|------|------|
| System Prompt + 规则 | ≤ 8K | ___ |
| Few-shot 示例 | ≤ 4K | ___ |
| 用户输入（单次最大） | ≤ 100K | ___ |
| 输出预留 | ≥ 20K | ___ |
| 缓冲区 | ≥ 8K | ___ |
```

任何规则新增时，必须在此表中更新实际占用量。若实际 > 预算，必须删减或拆分。

**上下文压缩原则：**
1. 规则用祈使句，不用解释性语句。 ✅ "输出 JSON，不要 Markdown"  ❌ "因为下游系统需要 JSON 格式，所以请……"
2. 示例优于规则。一个 50 token 的示例 > 一条 200 token 的规则
3. 规则模块只在需要时加载，用完即卸载

**上下文健康检查：**
```
"请报告本次运行的上下文使用情况：
- 输入总 token 数（估算）
- 规则模块实际加载了哪些
- 是否有规则被输入数据挤出了有效注意力范围"
```
