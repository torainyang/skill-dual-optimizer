## 框架诊断示例：Generator 模式 — `report-generator`

# 审查结论

## 核心定性
原型级，最大硬伤：确认门槛缺失导致可能用默认数据源生成错误报告

## 问题清单
- 〔D1 执行可靠性〕P1 description 只写了"生成报告"，缺少触发关键词（"周报""月报""数据汇总"），导致触发率低
- 〔D1 执行可靠性〕P1 SKILL.md 第 45-80 行的报告模板占主文件 30%，应拆到 `assets/report-template.md`——关键规则被埋在中间
- 〔D1 执行可靠性〕P2 Gotchas 为空——裸奔，LLM 容易在未确认数据范围时直接生成
- 〔D2 决策质量〕P0 工作流缺少"数据源确认"步骤，用户未指定数据源时直接用默认值，可能产出错误报告
- 〔D2 决策质量〕P2 输出报告缺少"生成时间"和"数据截止日期"字段，影响可用性

## 优化计划
1. 修改 `SKILL.md` frontmatter — description 改为"根据指定数据源生成结构化报告。当用户提到'生成报告''周报''月报''数据汇总''报表'时使用。" / 提升触发准确性 / 仅本文件 / 否
2. 新建 `assets/report-template.md` — 将 SKILL.md 第 45-80 行的报告模板移入 / 减少主文件篇幅，关键规则上移 / 需同步修改 SKILL.md / 否
3. 修改 `SKILL.md` 工作流 — 在 Phase 1 后增加"确认数据源和时间范围"，设为硬性门槛 / 防止用默认数据源生成错误报告 / 仅本文件 / 否
4. 修改 `SKILL.md` Gotchas — 增加"不要在用户未确认数据范围前生成最终报告" / 补充防护 / 仅本文件 / 否

## 专家组诊断示例 — `skill-xxx`

# 审查结论

## 核心定性
框架级，最大硬伤：按需加载触发条件无法覆盖核心使用场景，导致 Quick-Fail 检查被跳过

## 共识（3个专家均指出）
- 〔D1 执行可靠性〕P0 quality-rubric 按需加载条件"打分/评分"无法覆盖 Phase 1 快速判断场景，Quick-Fail 检查不可执行
- 〔D1 执行可靠性〕P1 重复三原则定义在 D1 扫描深处，Phase 3/4 执行时遗忘风险高
- 〔D2 决策质量〕P0 Phase 4 Verify 只验证结构属性，未验证问题是否被修复

## 分歧

| 问题 | 认知负荷分析师 | Prompt架构师 | 判断 |
|------|-------------|------------|------|
| D1/D2 扫描清单是否拆到 references | 拆——减少主文件体积 | 不拆——审查时必须逐项对照，拆出增加延迟 | 采纳不拆：扫描清单是运行时必需，拆出不减少认知负荷 |

## 补充（仅1个专家提出但高价值）
- 〔D1 执行可靠性〕P1 字数统计让 LLM 做确定性计算，自身违反 LLM vs 脚本原则 | 来源：Prompt架构师
- 〔D2 决策质量〕P1 Quick-Fail 后路径缺失，跳过深度审查后无回流补扫机制 | 来源：流程完备性审计师

## 优化计划
1. 修改 `SKILL.md` Phase 1 按需加载 — quality-rubric 触发条件改为"Phase 1 快速判断阶段必读" / 修复 Quick-Fail 不可执行 / 仅本文件 / 否
2. 修改 `references/prompt-engineering-principles.md` — 新增 H5 重复三原则 / 规则唯一定义点 / 需同步修改 SKILL.md D1 扫描 / 否
3. 修改 `SKILL.md` Phase 4 — 增加问题修复验证、改动面一致性、Quick-Fail 复检 / 修复验证缺失 / 仅本文件 / 否

请确认是否执行。
