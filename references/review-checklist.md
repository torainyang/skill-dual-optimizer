# Skill Review Checklist

按需审查，不必每次穷举所有项目。若用户已指定优化方向，先覆盖对应部分。

## 审查视角

每次审查时，从两个互补视角分别扫描，避免单一视角盲区：

### 视角 A: LLM 执行质量（Prompt 在模型眼里是否高效可靠）

重点关注：
- 关键约束是否在高注意力区（前 20% / 后 20%）
- 是否存在幻影指令、指令冲突、模糊边界
- 正向示例与负向禁止的比例是否 ≥ 3:1
- Token 是否浪费（语义重复、格式开销、不必要的常识）
- 规则是否可客观验证

### 视角 B: 用户价值质量（Skill 对终端用户是否真正有用）

重点关注：
- 工作流步骤是否合理、有序、无遗漏
- 输出是否可直接使用，格式是否稳定
- 边缘输入（极短、极长、模糊、多主题）是否有处理
- 用户是否需要做不必要的工作（手动拆分、说"继续"等）
- 交互是否高效，onboarding 是否简洁

两个视角的发现分别标注来源（A 或 B），在 Plan 中按影响力排序，不按视角分组。

---

## 1. Triggering 〔A+B〕

- `name` 是否只含小写字母、数字、连字符 〔A〕
- 目录名是否与 `name` 完全一致 〔A〕
- `description` 是否同时说明"做什么"和"何时使用" 〔B〕
- `description` 是否包含用户真实会说的关键词 〔B〕
- `description` 是否避免第一人称和空泛描述 〔A〕
- `description` 是否容易让模型判断是否应触发 〔A〕
- `description` 是否写了必要的不适用场景 〔B〕

常见优化：

- 把"适用场景"里的触发词回收到 `description`
- 去掉模糊描述，例如"帮助处理各种内容"
- 把技能边界写清楚，避免和相邻 skill 抢触发

## 2. Workflow 〔A+B〕

- 是否有明确顺序 〔B〕
- 是否存在必须确认却没有等待确认的步骤 〔B〕
- 是否说明输入、输出、落盘位置或交付格式 〔B〕
- 是否对失败场景给出降级策略 〔B〕
- 是否把"应该做"和"禁止做"写清楚 〔A〕
- 是否存在幻影指令（LLM 无法执行的指令，如"默默验证后再输出"） 〔A〕
- 是否存在互相矛盾的指令 〔A〕

常见优化：

- 增加 checklist
- 把"先检查再执行"写成硬性门槛
- 为关键输出增加固定格式

## 3. Progressive Disclosure 〔A〕

- `SKILL.md` 是否只保留核心流程 〔A〕
- 详细规则、示例、长说明是否拆到 `references/` 〔A〕
- `references/` 是否都直接从 `SKILL.md` 链接 〔A〕
- 是否避免多层嵌套引用 〔A〕
- 是否避免在 `SKILL.md` 和 `references/` 重复大段内容 〔A〕
- 关键约束是否放在 SKILL.md 的前 20% 或后 20%（高注意力区） 〔A〕
- 超过 100 行的 reference 文件是否有 TOC 〔A〕

常见优化：

- 把大段示例、规则表、术语表移到 `references/`
- 在 `SKILL.md` 里只写"什么时候读哪个 reference"

## 4. Resource Strategy 〔A+B〕

- 是否存在反复手写且适合脚本化的操作 〔B〕
- 是否缺少稳定执行所需的模板、脚本或参考资料 〔B〕
- `assets/` 是否只放输出资源，不放说明文档 〔A〕
- 是否创建了不必要的 README、CHANGELOG 等噪音文件 〔A〕

常见优化：

- 把重复命令封装到 `scripts/`
- 删除与 skill 执行无关的辅助文档

## 5. Gotchas & Guardrails 〔A〕

- 是否有高信号的常见坑点 〔A〕
- 是否记录模型最容易犯的错 〔A〕
- 是否提醒误触发、误判、默认行为过强等问题 〔A〕
- Gotchas 是不是随着使用在增长，还是根本没有 〔A〕
- 是否写了显而易见的废话（重复模型本来就知道的常识） 〔A〕

常见优化：

- 把重点放在"你容易做错什么"，而不是"这是一个很厉害的 skill"
- 用正向示例替代"不要做X"的规则

## 6. Output Contract 〔B〕

- 输出是否可直接给用户或下游 agent 使用 〔B〕
- 是否区分"审查结论""优化计划""最终修改结果" 〔B〕
- 是否说明确认回复的触发条件，例如"确认""开始修改" 〔B〕
- 是否在最终汇报中列出文件和剩余风险 〔B〕
- 输出长度是否有软性控制，避免过度生成 〔A〕
- 输出是否有足够多样性，不总是同一模式 〔B〕

常见优化：

- 固定 plan 模板
- 固定最终汇报模板
- 把用户可选回复写清楚

## 7. Setup / Config / Memory 〔A+B〕

- skill 是否有首次使用需要确认的配置 〔B〕
- 是否应该用配置文件或稳定目录保存状态 〔B〕
- 是否每次都让模型重复问同样的问题 〔A〕
- 这个 skill 是否适合沉淀历史记录 〔B〕
- 是否可以通过日志或状态文件提升稳定性 〔A〕

## 8. Distribution & Reuse 〔B〕

- 这是一个仅个人使用的 skill，还是适合公开分发的 skill 〔B〕
- 如果公开分发，是否把个人私有路径 / 偏好 / 环境依赖写死了 〔B〕
- 是否存在 railroading 或过度设计 〔A〕
- 是否为简单场景引入过多结构 〔A〕

> 问题优先级排序规则见 SKILL.md Step 3。
